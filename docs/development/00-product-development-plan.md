## Decision

The master specification is frozen for implementation start. Do not keep expanding it before Phase 0. New learnings go into implementation notes or issue backlog unless they change product boundaries, permissions, data model, or external data flows.

## Development Strategy

Build the product from the safest inner core outward:

1. Foundation: WXT, MV3, TypeScript, React, test harness.
2. Local contracts: typed models, Dexie schema, migrations, settings.
3. HH read layer: vacancy parser, fixtures, passive status extraction.
4. Local value: tracker, status history, scoring, export.
5. UI shell: popup, side panel, dashboard.
6. AI: payload preview, redaction, cache, Strict Privacy.
7. Cover letters: generation, edit, save, copy.
8. n8n: opt-in webhook, HMAC, retry, event log.
9. Release hardening: privacy/safety tests, manifest audit, manual QA.

Search badges, queue, guided apply, HR communication, and multi-site support are deliberately out of Phase 0/1 until the Core is stable.

## Working Model

One iteration should be small enough for one focused AI autopilot run and one human review.

Recommended unit:

```text
one iteration -> one branch or local change set -> one commit -> one push
```

For now, commits can go to `main` while the project is private and early. Once implementation begins, prefer short-lived branches:

```text
codex/iter-001-wxt-scaffold
codex/iter-002-domain-models
```

## Definition Of Ready

An iteration is ready for autopilot when it has:

- a narrow goal;
- allowed files or target folders;
- explicit non-goals;
- safety constraints;
- acceptance criteria;
- validation commands;
- expected commit message.

## Definition Of Done

Every implementation iteration must satisfy:

- no forbidden HH automation;
- no broad permissions;
- no secrets or keys committed;
- tests or fixture checks where practical;
- typecheck/build green once the project has scripts;
- docs updated if contracts or behavior changed.

## 8. Scoring Model (Rule-based)

Алгоритм выполняется локально на `Content Script` сразу после парсинга данных, без обращения к сети. Итоговый балл `RuleScore` варьируется от `0` до `100`.

### Компоненты расчета:

1. **Проверка Черных Списков (Стоп-фактор):**
* Если `Company.isBlacklisted === true` $\rightarrow$ `RuleScore = 0`, статус автоматически меняется на `blacklist`.
* Если в тексте вакансии (`descriptionText`) или заголовке найдены глобальные стоп-слова пользователя (например: *"холодные звонки"*, *"crypto scam"*, *"1С"* для frontend-разработчика) $\rightarrow$ `RuleScore = 0`.


2. **Матчинг Ключевых Навыков (Вес: 40% / макс. 40 баллов):**
* Пусть $S_{vac}$ — множество навыков из вакансии, $S_{prof}$ — множество ключевых навыков выбранного профиля кандидата.
* Формула: $Score_{skills} = \left( \frac{|S_{vac} \cap S_{prof}|}{|S_{vac}|} \right) \times 40$
* *Примечание:* Если в вакансии не указаны Hard Skills в виде тегов, производится поиск по ключевым словам из $S_{prof}$ в тексте `descriptionText` (каждое совпадение добавляет по 5 баллов, в пределах 40).


3. **Матчинг Заголовка Вакансии (Вес: 30% / макс. 30 баллов):**
* Поиск вхождений строк из `Profile.targetTitles` в заголовок `Job.title` (без учета регистра).
* Полное совпадение или вхождение сильной маски (например, "React Developer" в профиле и "Senior React Developer" в вакансии) $\rightarrow$ 30 баллов.
* Частичное совпадение (хотя бы одно ключевое слово, например "Developer") $\rightarrow$ 15 баллов.
* Нет совпадений $\rightarrow$ 0 баллов.


4. **Финансовый Матчинг (Вес: 30% / макс. 30 баллов):**
* Если вилка зарплаты в вакансии не указана $\rightarrow$ дефолтное значение **15 баллов** (нейтральный фактор).
* Если указана:
* `Job.salaryMax` < `Profile.preferredSalaryMin` $\rightarrow$ 0 баллов.
* `Job.salaryMin` $\ge$ `Profile.preferredSalaryMin` $\rightarrow$ 30 баллов.
* `Profile.preferredSalaryMin` находится внутри интервала [`salaryMin`, `salaryMax`] $\rightarrow$ 25 баллов.

* **Принцип наименьших привилегий (MV3 Permissions):**
В файле `manifest.json` запрашиваются минимально необходимые доступы:
* `storage` — для сохранения конфигураций расширения.
* `activeTab` — доступ к содержимому вкладки предоставляется *только* в момент явного открытия пользователем страницы вакансии (снижает уровень тревожности ИБ-систем Chrome Web Store).
* `host_permissions` ограничиваются строго пулом целевых адресов: `*://*.hh.ru/*`, `*://*.hh.ua/*`, `https://api.openai.com/*`, `https://api.deepseek.com/*`, `https://openrouter.ai/*`.


* **Безопасность API-ключей:** Ключи пользователей сохраняются исключительно в `chrome.storage.local`. Синхронизация через `chrome.storage.sync` отключена, чтобы исключить передачу сырых токенов авторизации через облачные сервера синхронизации Google.
* **Минимизация данных перед отправкой в LLM:**
Скрипт контента извлекает только значимый текст. Исходный HTML-код страницы полностью очищается с помощью регулярных выражений или легковесных парсеров от лишних скриптов, рекламных блоков, сайдбаров HH.ru и футеров. Персональные данные пользователя (кроме тех, что он осознанно внес в локальный менеджер профилей) никогда не передаются в LLM.
* **Защита n8n Webhook:** payload вебхука отправляется методом `POST`. Расширение предоставляет поле ввода для кастомизируемого секретного заголовка (например, `X-CareerSignal-Token` или `Authorization: Bearer <token>`), позволяя пользователю защитить приемник своего инстанса n8n от стороннего спама.

## 12. Risks and Mitigations (Противодействие антибот-системам)

HH.ru защищен комплексами Qrator Labs и жестко пресекает автоматический парсинг (парсинг страниц без участия пользователя быстро приводит к капче или временному бану по IP).

| Риск | Причина | Стратегия предотвращения (Mitigation) |
| --- | --- | --- |
| **Блокировка за частые фоновые HTTP-запросы** | Массовый фетчинг страниц вакансий из поисковой выдачи через `fetch(url)` из расширения. | **Полный запрет фонового парсинга.** Расширение извлекает данные *исключительно* в момент, когда пользователь физически открыл вкладку вакансии. Нет фоновых сетевых запросов к HH — нет триггеров для антифрода. |
| **Поломка парсера из-за обновлений верстки** | Изменение CSS классов, верстка структуры блоков, внедрение обфускации классов (A/B тесты HH). | **Парсинг глобального состояния.** Вместо DOM-элементов расширение считывает контент скрипта `<script id="hh-state" type="application/json">`. Структура JSON-данных бэкенда стабильна и меняется крайне редко. |
| **Детекция эмуляции кликов антифродом** | Попытка программно нажать кнопку отклика через `document.querySelector('.apply-btn').click()`. | **Исключение автоматических кликов.** Процесс разделен: соискатель сам нажимает нативную кнопку отклика. Расширение лишь перехватывает появление формы через `MutationObserver` и подставляет текст в буфер или поле ввода. |
| **React State Desynchronization** | Прямая программная вставка текста в `textarea.value = text` приводит к тому, что внутренний React-стейт формы HH.ru не видит изменений, и кнопка отправки остается заблокированной. | **Симуляция нативных событий.** Вставка текста производится через цепочку событий: `textarea.focus()`, программная запись в буфер и вызов `document.execCommand('paste')` или через последовательную генерацию `InputEvent` с флагами `{ bubbles: true }`. |

# От ботов до ассистента: Стратегия безопасного создания расширения для HeadHunter без риска блокировки

## Анализ концепции: выявление ключевых рисков и противоречий

Настоящий раздел представляет собой глубокий критический анализ исходной концепции браузерного расширения CareerSignal HH Copilot. Цель анализа — выявить скрытые риски, внутренние противоречия и потенциальные слабые места, которые могут препятствовать успешному запуску и дальнейшему развитию продукта. Анализ основан на предоставленных материалах и сфокусирован на трех ключевых областях: соответствие условиям использования (ToS) HeadHunter, техническая реализуемость и безопасность.

Первоочередной и наиболее критическим риском является столкновение намеченной функциональности с политиками hh.ru. Исходная концепция прямо указывает на запрет парсинга, скрапинга или их аналогов [[41]]. Любая автоматизация извлечения данных со страниц вакансий и поисковой выдачи hh.ru по своей сути является формой скрапинга. Хотя судебная практика показывает, что сбор публично доступной информации может быть легальным, даже если это формально нарушает условия использования [[21,23]], такое поведение неизбежно привлекает внимание антибот-систем. Эти системы используют многофакторный анализ для определения автоматизированной активности, включая проверку заголовков HTTP, сигналов из JavaScript-окружения (например, наличие `headless`), анализ DOM-операций и поведенческие паттерны пользователя [[31,37,39]]. Таким образом, сам факт того, что расширение будет извлекать данные с hh.ru, уже создает базовый уровень риска. Однако этот риск значительно возрастает при добавлении функционала полуавтоматического отклика, заложенного в планах на MVP-3 и MVP-4.

Второе серьезное противоречие заключается в сочетании цели «не спам-бот автооткликов» с планируемым функционалом полуавтоматической отправки. Функция, которая готовит текст письма и подставляет его в форму отклика на hh.ru, даже если финальное нажатие кнопки «Отправить» требует действия пользователя, находится на грани политики сайта. Такие действия могут быть классифицированы как использование сторонних инструментов для модификации функционала сайта [[2]], что явно запрещено LinkedIn и, скорее всего, также применяется к hh.ru. Антибот-системы могут обнаружить не только IP-адрес, но и уникальные JS-объекты, мониторинг DOM-элементов и последовательности событий, характерные для автоматизированных действий [[31]]. Подстановка текста через JavaScript (`element.value = text`) является одним из таких легко детектируемых сигналов [[39]]. Следовательно, такой подход несет в себе высокий риск быстрой блокировки как самого расширения, так и аккаунтов его пользователей. Этот риск прямо противоречит главной цели проекта — создание безопасного инструмента.

Третье существенное противоречие связано с использованием API-ключей для ИИ (BYOK - Bring Your Own Key). Предложение использовать OpenAI, DeepSeek или другие сервисы напрямую из браузера сопряжено с критической уязвимостью. Хранение API-ключей в клиентском коде JavaScript равносильно их открытой публикации. Любой пользователь сможет получить доступ к этим ключам через инструменты разработчика в браузере. Это приведет к двум негативным последствиям: во-первых, бесконтрольные расходы на API-запросы, что сделает проект экономически неустойчивым; во-вторых, возможность для злоумышленников использовать эти ключи для своих целей, что может привести к блокировке учетной записи API у провайдера и финансовым потерям для разработчиков [[50]]. Более того, некоторые фреймворки, такие как Transformers.js, могут быть отклонены в Chrome Web Store за включение удаленно размещенного кода, что усложняет их использование в расширениях [[163,169]].

Четвертое противоречие возникает между требованием локального хранения данных и необходимостью отправки событий во внешний webhook (n8n). Требование «локального первенства» и экспорта CSV/JSON является сильной стороной с точки зрения приватности. Однако делегирование расширению права отправлять данные на произвольный сервер, задаваемый пользователем, создает потенциальную уязвимость. Если хранилище расширения будет скомпрометировано, злоумышленник получит возможность перенаправлять чувствительные данные пользователя (информация о просмотренных вакансиях, история поиска, даже сгенерированные ИИ письма) на сервер, контролируемый им. Хотя это не является прямым нарушением ToS hh.ru, это представляет серьезную угрозу безопасности самого пользователя. Эта функция должна быть реализована как полностью опциональная, с максимальной прозрачностью и контроля со стороны пользователя.

Наконец, пятая слабая сторона — это план по созданию гибкой системы профилей кандидата («Universal, AI Engineer / LLM / RAG...»). В то время как идея предоставить пользователю несколько вариантов резюме для разных ролей выглядит привлекательно, ее реализация в рамках MVP-концепции становится чрезмерно сложной. Создание и поддержка такого большого количества профилей потребует значительных усилий как со стороны разработчиков, так и со стороны пользователей. Для MVP гораздо эффективнее начать с одной базовой модели резюме и позволить пользователю вручную адаптировать его, возможно, с помощью советов от ИИ, вместо того чтобы пытаться автоматически генерировать десятки различных профилей.

| Риск | Описание | Вероятность | Влияние |
| :--- | :--- | :--- | :--- |
| **Нарушение ToS hh.ru** | Автоматическое извлечение данных (скрапинг) и попытка автоматизации действий (отклики) прямо противоречит условиям использования hh.ru, которые запрещают подобные методы [[41]]. | Высокая | Критическое (блокировка аккаунта пользователя и расширения) |
| **Обнаружение антибот-системами** | Использование JS для манипуляции DOM и подстановка текста в формы являются узнаваемыми сигнатурами ботов для современных антибот-систем (например, Cloudflare) [[31,39]]. | Высокая | Критическое (временная или постоянная блокировка IP/аккаунта) |
| **Утечка API-ключей** | Хранение API-ключей для ИИ в клиентском коде браузера делает их доступными для любого пользователя, что приведет к финансовым потерям и возможному бану аккаунта API [[50]]. | Высокая | Высокое (экономическая неустойчивость проекта) |
| **Уязвимость n8n интеграции** | Возможность отправки данных на произвольный пользовательский URL-адрес создает риск утечки данных, если расширение будет скомпрометировано. | Средняя | Высокое (нарушение конфиденциальности данных пользователя) |
| **Сложность реализации профилей** | Попытка реализовать сложную систему из множества профилей в MVP создаст избыточную сложность и замедлит разработку. | Высокая | Среднее (задержка выхода на рынок, высокие затраты на разработку) |

В совокупности эти риски и противоречия указывают на то, что исходная концепция, хотя и содержит ядро ценной идеи, слишком рискованна и содержит внутренние несоответствия для успешного запуска MVP. Необходимо кардинально пересмотреть стратегию, сместив акцент с автоматизации на анализ и организацию, а также внедрить архитектурные решения, обеспечивающие безопасность и соответствие правилам платформы.

## Улучшенная продуктовая стратегия и безопасный user-controlled подход

Для преодоления выявленных рисков и создании жизнеспособного продукта необходимо кардинально пересмотреть исходную продуктовую стратегию. Вместо того чтобы стремиться к максимальной автоматизации, как это часто бывает в инструментах для поиска работы, новая стратегия должна быть сосредоточена на двух ключевых принципах: **полный пользовательский контроль** и **помощь в принятии решений**, а не в выполнении действий. Это позволит сохранить ценность для пользователя, избегая при этом прямого конфликта с политиками hh.ru и антибот-системами.

Основная ценность нового продукта заключается не в том, чтобы заменить пользователя, а в том, чтобы стать его «ассистентом» или «советником». Расширение должно предоставлять пользователю всю необходимую информацию, анализ и инструменты для того, чтобы он мог сделать наилучший возможный выбор самостоятельно. Этот shift в философии позволяет отказаться от всех видов автоматических действий, которые были идентифицированы как высокорискованные. Полностью исключаются любые функции, автоматически подставляющие текст в форму отклика, автоматически нажимающие кнопки или выполняющие какие-либо другие действия без прямого и однозначного подтверждения пользователя. Даже полуавтоматический отклик, который был заложен в планах, должен быть либо полностью исключен из MVP, либо переосмыслен. Вместо подстановки сгенерированного текста в форму, более безопасной и пользовательской альтернативой будет предоставление этого текста в виде готового к копированию блока. Пользователь должен сам вставить его в нужное поле. Это незначительно увеличивает количество кликов, но полностью устраняет сигналы для детекторов ботов и соответствует духу принципа «user-controlled».

Стратегия должна четко декомпозировать функционал на три уровня взаимодействия с пользователем:

1.  **Анализ (Analyze):** На этом уровне расширение выступает в роли «разведчика». Его единственная задача — безопасно извлекать информацию со страницы hh.ru, структурировать ее и сохранять локально. Он предоставляет пользователю данные для анализа, но не предпринимает никаких дальнейших действий. Все кнопки и действия на этом этапе направлены на сохранение и маркировку вакансии («Сохранить», «Просмотрено», «Сильная вакансия»).

2.  **Подготовка (Prepare):** На этом уровне расширение становится «подготовителем». Оно использует мощь ИИ для генерации качественных материалов, которые помогут пользователю в его коммуникации. Это включает в себя:
    *   Генерацию анализа соответствия («Сильные стороны», «Чего не хватает»).
    *   Создание черновиков сопроводительных писем.
    *   Формулирование возможных ответов на сообщения от HR.
    Все эти материалы должны быть представлены пользователю в виде текстовых блоков, готовых к копированию и вставке вручную. Никаких автоматических действий здесь быть не может.

3.  **Управление (Manage):** На этом уровне расширение работает как «менеджер» процесса поиска работы. Оно помогает пользователю организовывать свои усилия: отслеживать статусы откликов, вести историю взаимодействий с компаниями, создавать базу данных избранных вакансий и экспортить ее для дальнейшего анализа. Интеграция с n8n в этой парадигме должна рассматриваться как продвинутая функция для синхронизации данных с внешними системами управления проектами или CRM, а не как обязательный компонент.

Эта трехуровневая модель позволяет построить надежный и масштабируемый продукт. Она начинается с минимально необходимого набора функций (анализ), которые имеют самый низкий риск, и постепенно добавляет более сложные, но все еще безопасные возможности (подготовка, управление).

Важнейшим элементом стратегии является обеспечение максимальной прозрачности. Пользователь всегда должен понимать, что именно происходит и какие данные используются. Например, перед отправкой запроса на ИИ-анализ, интерфейс должен четко показывать, какой именно профиль кандидата и какие фрагменты описания вакансии будут переданы на обработку. Это не только повышает доверие, но и позволяет пользователю контролировать свою конфиденциальность. Перед отправкой данных на внешний ИИ-сервис необходимо реализовать механизм очистки от чувствительной личной информации (ФЛИ), такой как реальное имя, адрес электронной почты, номер телефона и т.д., которые могут содержаться в профилях кандидатов.

Кроме того, необходимо тщательно подойти к выбору технологического стека для ИИ. Прямое использование API-ключей из браузера недопустимо. Единственно правильным решением является внедрение промежуточного backend-сервиса (например, на AWS Lambda, Vercel Functions или Netlify Functions). Расширение будет отправлять запросы к этому сервису, который уже будет безопасно взаимодействовать с API крупных моделей (GPT-4, Claude Opus) [[118,121]]. Этот сервис может также выполнять роль прокси-сервера, позволяя пользователям подключать свои собственные ключи, которые хранятся на сервере и не доступны клиенту [[94]]. Это решение добавляет некоторую сложность в архитектуру, но является абсолютно необходимым для обеспечения безопасности и экономической устойчивости продукта.

Наконец, стратегия должна предусматривать поэтапный вывод продукта на рынок. Начинать следует с MVP, который фокусируется исключительно на анализе и локальном управлении. Функционал, связанный с ИИ и внешними интеграциями, должен быть реализован после того, как основная архитектура и пользовательский опыт будут отлажены и подтверждены на практике. Такой подход позволяет быстро запустить рабочий продукт, получить обратную связь от ранних пользователей и итерационно развивать его, добавляя новые функции по мере необходимости и снижая риски.

## Архитектура и технологический стек расширения

Выбор правильной архитектуры и технологического стека является фундаментом для создания стабильного, безопасного и масштабируемого браузерного расширения. Для проекта CareerSignal HH Copilot, ориентированного на Chromium-браузеры и первый релиз, следует выбрать современные и хорошо поддерживаемые технологии, которые соответствуют ограничениям Manifest V3.

**Основа: Manifest V3 и WXT**

Решение использовать Manifest V3 является стандартом де-факто для новых расширений в Chromium-браузерах, поскольку Google больше не принимает расширения на основе Manifest V2 [[46,82,158]]. Manifest V3 предлагает улучшенную безопасность и производительность за счет замены постоянных фоновых страниц на сервис-воркеры, которые активируются по событиям и затем завершают работу [[82,233]]. Это изменение требует иного подхода к архитектуре, но в долгосрочной перспективе является преимуществом. Для упрощения разработки на Manifest V3 рекомендуется использовать фреймворк WXT (Web Extension Tools) [[156]]. WXT предоставляет удобный инструментарий для сборки, отладки и публикации расширений, отлично интегрируясь с современными frontend-фреймворками и языками, такими как TypeScript и React [[229]]. Использование TypeScript для всего проекта (manifest, content scripts, popup, side panel) настоятельно рекомендуется для обеспечения строгой типизации, что значительно снижает количество ошибок времени выполнения и улучшает читаемость кода.

**Структура расширения**

Расширение должно состоять из следующих основных частей:
*   **`manifest.json`:** Основной файл, описывающий метаданные расширения, запрашиваемые разрешения и точки входа (popup, side panel, service worker) [[161]]. Для MVP потребуются разрешения `activeTab` для доступа к текущей вкладке и `storage` для локального хранения настроек [[15]].
*   **Service Worker:** Замена фоновой страницы. Он будет отвечать за обработку системных событий, таких как отправка webhook-запросов, и может служить точкой входа для взаимодействия с backend-сервисом.
*   **Content Script:** JavaScript-код, который встраивается в страницы hh.ru. Его задача — извлекать данные с веб-страницы, модифицировать DOM (добавлять кнопки, бейджи) и взаимодействовать с остальной частью расширения через `runtime.sendMessage`.
*   **Popup UI:** Минималистичный интерфейс, открывающийся по клику на иконку расширения. Он должен содержать краткую сводку (например, количество новых вакансий) и ссылку на основную панель управления.
*   **Side Panel UI:** Основная рабочая область расширения, где пользователь будет видеть историю всех просмотренных и сохраненных вакансий, их статусы и управлять ими. Использование Side Panel предпочтительнее, чем расширение Popup'а, так как он предоставляет больше места для интерфейса и лучше подходит для управления списком данных.
*   **Backend Service:** Простой серверный компонент (например, на Node.js с Express или Python), который будет обрабатывать запросы к API ИИ. Он будет выполнять роль прокси, скрывая API-ключи и обеспечивая безопасную связь между расширением и LLM.

**Хранилище данных: IndexedDB vs localStorage**

Требование «локального первенства» требует тщательного выбора механизма хранения. Для хранения небольших объемов данных, таких как пользовательские настройки, можно использовать `chrome.storage.local` [[161]]. Однако для основной базы данных, содержащей множество записей о вакансиях, компаниях, профилях и откликах, более подходящим решением является IndexedDB [[197]]. IndexedDB — это мощная низкоуровневая API для клиентского хранения значительных объемов структурированных данных, включая файлы и блобы [[83]]. В отличие от `localStorage`, который ограничен несколькими мегабайтами и работает с простыми строками, IndexedDB позволяет хранить сложные объекты, создавать индексы для быстрой сортировки и поиска, и работает асинхронно, не блокируя основной поток пользовательского интерфейса [[198,201]]. Для упрощения работы с IndexedDB в проекте рекомендуется использовать библиотеку-обертку, такую как Dexie.js [[77]]. Важно помнить, что в Manifest V3 операции с IndexedDB из сервис-воркера напрямую невозможны; все запросы к базе данных должны выполняться из контекста Content Script или UI-пользовательского интерфейса через асинхронные сообщения [[78,79]].

**Взаимодействие с ИИ**

Как было установлено ранее, прямое обращение к API ИИ из браузера недопустимо. Поэтому архитектура должна включать промежуточный backend-сервис. Расширение будет отправлять POST-запросы на эндпоинт своего сервера, передавая туда необходимые данные (например, описание вакансии и профиль кандидата). Сервер, получив запрос, будет использовать свой собственный API-ключ для взаимодействия с LLM (OpenAI, Anthropic и т.д.) и возвращать результат обратно в расширение. Этот подход обеспечивает безопасность ключей и позволяет легко менять провайдеров ИИ в будущем. Для передачи данных можно использовать open-source решения, такие как `hhcli` или `hh-applicant-tool`, которые демонстрируют локальное хранение информации и использование LLM для оценки вакансий [[87,88,94]].

**Интеграция с n8n**

Интеграция с n8n через вебхук должна быть реализована как опциональная функция. Пользователь должен иметь возможность ввести свой URL-адрес в настройках расширения (хранящийся в `chrome.storage.local`). При срабатывании события (например, «отправлен отклик»), расширение будет собирать JSON-объект с информацией о вакансии и статусе, и отправлять его на указанный URL-адрес с помощью `fetch` API. Чтобы обеспечить безопасность, интерфейс настроек должен показывать пример JSON-тела, которое будет отправлено, и давать пользователю возможность отключить эту функцию полностью.

Таким образом, предлагаемая архитектура является модульной, безопасной и соответствует современным стандартам разработки расширений. Она позволяет реализовать все необходимые функции MVP, минимизируя при этом риски, связанные с безопасностью и соответствием политикам браузеров.

## Разработка MVP: модель данных, система оценки и структура UX

Эффективная разработка MVP требует четкого определения модели данных, простой, но работающей системы оценки и интуитивно понятной структуры пользовательского интерфейса. Эти три компонента составляют ядро функциональности расширения.

**Модель данных**

Для организации информации о вакансиях, кандидатах и их взаимодействиях необходимо определить четкие структуры данных. Все эти структуры будут храниться в IndexedDB.

*   **`Job` (Вакансия):** Основная сущность, представляющая одну вакансию.
    *   `id: string`: Уникальный идентификатор вакансии с hh.ru (например, `HH:<vacancy_id>`). Является первичным ключом.
    *   `sourceUrl: string`: Полный URL-адрес страницы вакансии на hh.ru.
    *   `title: string`: Название вакансии.
    *   `company: string`: Название компании-работодателя.
    *   `salary: string | null`: Зарплатное предложение.
    *   `city: string | null`: Город.
    *   `typeOfWork: 'remote' | 'hybrid' | 'office' | null`: Тип занятости (удаленно, гибрид, офис).
    *   `experience: string | null`: Требуемый опыт.
    *   `descriptionText: string`: Очищенный текстовое описание вакансии.
    *   `skills: string[]`: Массив ключевых навыков, извлеченных из описания.
    *   `status: JobStatus`: Текущий статус вакансии.
    *   `profileId: string | null`: Идентификатор выбранного профиля резюме.
    *   `matchScore: number | null`: Результат правила-базированной оценки.
    *   `aiAnalysis: string | null`: Сгенерированный ИИ-текст с анализом соответствия.
    *   `coverLetter: string | null`: Сгенерированный ИИ-черновик сопроводительного письма.
    *   `createdAt: Date`: Дата и время создания записи.
    *   `updatedAt: Date`: Дата и время последнего обновления.

*   **`Profile` (Профиль):** Сущность, представляющая резюме кандидата.
    *   `id: string`: Уникальный идентификатор профиля.
    *   `name: string`: Название профиля (например, "AI Engineer", "CRM Integrator").
    *   `resumeText: string`: Полный текст резюме.
    *   `skills: string[]`: Массив ключевых навыков из резюме.
    *   `lastUsedAt: Date | null`: Дата последнего использования этого профиля.

*   **`Application` (Отклик):** Сущность для отслеживания истории откликов.
    *   `id: string`: Уникальный идентификатор отклика.
    *   `jobId: string`: Идентификатор связанной вакансии.
    *   `appliedAt: Date`: Дата отправки отклика.
    *   `statusHistory: Array<{ status: ApplicationStatus, date: Date }>`: История изменения статуса отклика.

*   **`Event` (Событие):** Логирование важных событий для отправки во внешние системы.
    *   `id: string`: Уникальный идентификатор события.
    *   `type: 'application_sent' | 'hr_replied' | 'strong_vacancy_found' | 'daily_summary'`: Тип события.
    *   `payload: object`: Данные, связанные с событием (например, объект `Job`).
    *   `sentAt: Date | null`: Дата и время отправки webhook.

**Rule-Based Scoring Model**

Для MVP-1 требуется простая, но объективная система оценки. Она должна работать быстро и не требовать вызова ИИ. Можно реализовать модель, которая начисляет баллы за совпадения и вычитает за несоответствия.

Пример системы оценки на основе правил:

| Фактор | Описание | Баллы (+) | Штрафы (-) |
| :--- | :--- | :--- | :--- |
| **Совпадение по навыкам** | Наличие ключевого слова из профиля в массиве `Job.skills`. | +10 |
| **Совпадение в описании** | Наличие ключевого слова из профиля в тексте `Job.descriptionText`. | +5 |
| **Совпадение по компании** | Наличие названия компании из профиля в `Job.company`. | +3 |
| **Требуемые технологии** | Наличие в `Job.descriptionText` слов "требуются", "знание", "умение" рядом с технологиями из профиля. | +2 за каждую найденную |
| **Запрещенные термины** | Наличие в `Job.descriptionText` слов "без опыта", "не нужен", "не требуется" рядом с технологиями из профиля. | -10 за каждое найденное |
| **Уровень опыта** | Соответствие строки `Job.experience` ("от X лет", "до X лет") диапазону опыта из профиля. | +5 (соответствие), -5 (несоответствие) |

Общий балл суммируется и нормализуется до 100-балльной шкалы. Этот балл используется для автоматической классификации вакансии (например, Match >= 85 — "сильная вакансия", Match < 50 — "низкий приоритет"). Эта система проста в реализации и дает пользователю быструю обратную связь о потенциальной релевантности вакансии.

**Структура пользовательского интерфейса**

UI должен быть минималистичным, интуитивно понятным и не мешать основному опыту пользователя на hh.ru.

*   **Content Script (На странице вакансии):** Расширение добавляет в правый верхний угол страницы вакансии небольшую панель действий.
    *   Кнопка «Сохранить»: Добавляет вакансию в IndexedDB со статусом «new».
    *   Кнопка «Просмотрено»: Изменяет статус вакансии на «просмотрено».
    *   Кнопка «Сохранить»: Сохраняет вакансию в базу данных.
    *   Кнопка «Сильная вакансия»: Устанавливает статус «сохранено» и запускает ИИ-анализ.
    *   Бейдж (на странице вакансии): Показывает правило-базированный балл совпадения рядом с кнопкой «Сохранить».

*   **Popup (По клику на иконку):** Открывает небольшое окно с краткой информацией.
    *   Заголовок: "CareerSignal HH Copilot".
    *   Бейдж: Общее количество новых (не просмотренных) вакансий в базе данных.
    *   Кнопка: "Открыть панель" -> переход на Side Panel.

*   **Side Panel (Основная рабочая область):** Это центр управления расширением.
    *   **Фильтры и поиск:** Панель содержит строки для фильтрации вакансий по статусу (все, просмотрено, сохранено, откликнутось), по профилю и общую строку поиска по названию, компании или навыкам.
    *   **Список вакансий:** В центре панели отображается список всех сохраненных вакансий, отсортированных по дате добавления. Каждая вакансия представлена карточкой.
    *   **Карточка вакансии:**
        *   **Заголовок, Компания, Зарплата.**
        *   **Бейдж с оценкой:** Цветной бейдж с числовым значением match-score (например, зеленый для >80, желтый для 50-80, красный для <50).
        *   **Статусный индикатор:** Иконка глаза (просмотрено), сердца (сохранено), галочки (откликнулся).
        *   **Кнопки действий:**
            *   "AI-анализ": Запускает генерацию анализа ИИ.
            *   "Сгенерировать письмо": Открывает модальное окно с опциями профилей и генерирует черновик письма.
            *   "Добавить в очередь": Добавляет вакансию в очередь откликов (для MVP-4).
            *   "Удалить": Удаляет вакансию из базы данных.

*   **Настройки (Settings Page):** Доступна через Side Panel. Позволяет пользователю:
    *   Управлять своими профилями (создавать, редактировать, удалять).
    *   Вводитьлючает три раздела: «Сильные стороны», «Чего не хватает в резюме», «Почему стоит откликнуться?». Пользователь видит точный payload перед отправкой.
    *   **Экспорт данных:** Возможность экспорта всех сохраненных вакансий в CSV/JSON. Экспорт включает только структурированные поля и `descriptionText`; полный HTML и скрытые метаданные исключены.
    *   **n8n Webhook (опционально):** Настройка в popup-меню. Пользователь вводит URL, подтверждает его дважды и видит пример JSON-сообщения, которое будет отправлено при событии «Новая вакансия просмотрена». Функция отключена по умолчанию.

*   **Что НЕ входит в MVP-1:**
    *   Генерация сопроводительных писем.
    *   Работа с поисковой выдачей (бейджи, подсветка).
    *   Полуавтоматический отклик или подстановка текста в форму.
    *   Работа с откликами, приглашениями и чатами.
    *   Локальное выполнение ИИ-моделей в браузере (Transformers.js и т.п.), так как это несовместимо с ограничениями Manifest V3 и требует значительных ресурсов [[163,169]].

### MVP-2: «The Organizer» (Организатор)

Этот MVP расширяет ценность, позволяя пользователю управлять всей своей историей поиска работы в одном месте, не покидая браузер.

*   **Целевая ценность:** Пользователь получает централизованное рабочее пространство для просмотра, фильтрации и управления всеми своими вакансиями.
*   **Обязательные функции:**
    *   **Side Panel (Боковая панель):** Полноценный интерфейс, встраиваемый в правую часть страницы hh.ru. Отображает все вакансии из IndexedDB в виде карточек. Каждая карточка показывает название, компанию, зарплату, бейдж с rule-based score и статусный индикатор.
    *   **Бейджи в поисковой выдаче:** Контент-скрипт добавляет небольшой цветной бейдж (например, зеленый для «Сильной», желтый для «Средней») на каждую карточку вакансии в результатах поиска hh.ru. Это позволяет пользователю быстро оценить релевантность, не открывая каждую вакансию.
    *   **Подсветка карточек:** Карточки в поисковой выдаче подсвечиваются в соответствии с их статусом (`new`, `viewed`, `saved`, `rejected_by_me`), используя CSS-классы.
    *   **Менеджер профилей:** Интерфейс в popup-меню для создания, редактирования и удаления профилей (`Profile`). Каждый профиль содержит имя, текст резюме и список ключевых навыков. Профили сохраняются в `chrome.storage.local`.
    *   **Quick Actions:** Кнопки на каждой карточке в Side Panel: «AI-анализ», «Сохранить», «Отклонить», «Удалить». Нажатие на «Сохранить» меняет статус на `saved`, а на «Отклонить» — на `rejected_by_me`.

*   **Что НЕ входит в MVP-2:**
    *   Генерация писем.
    *   Очередь откликов.
    *   Работа с разделом «Отклики» на сайте hh.ru.

### MVP-3: «The Assistant» (Помощник)

Этот MVP добавляет функцию, которая превращает расширение из трекера в настоящего помощника — генерацию персонализированных сопроводительных писем.

*   **Целевая ценность:** Пользователь получает готовый, адаптированный черновик письма, который он может скопировать и вставить в форму hh.ru, экономя 10–15 минут на каждую вакансию.
*   **Обязательные функции:**
    *   **Генерация сопроводительного письма:** Кнопка «Сгенерировать письмо» в Side Panel и на странице вакансии. При нажатии расширение отправляет данные о вакансии и выбранном профиле на backend-сервис. Сервис генерирует письмо, используя промпт, который акцентирует внимание на совпадениях и сильных сторонах.
    *   **Шаблоны писем:** Пользователь может выбрать один из предустановленных шаблонов: «HH стандарт», «Telegram (кратко)», «EN version». Каждый шаблон определяет структуру и стиль, но не меняет содержание.
    *   **Настройки генерации:** Пользователь может включить/выключить опции: «Без emoji», «Без markdown», «До 500 символов». Эти настройки передаются в промпт для LLM.
    *   **Сохранение письма:** Сгенерированное письмо сохраняется в поле `Job.cover_letter` в IndexedDB. Пользователь может просмотреть его в Side Panel и скопировать в буфер обмена одной кнопкой «Копировать».
    *   **Рекомендация резюме:** На основе анализа вакансии и имеющихся профилей расширение предлагает пользователю выбрать наиболее подходящий профиль для отклика.

*   **Что НЕ входит в MVP-3:**
    *   Автоматическая подстановка текста в форму hh.ru.
    *   Работа с очередью откликов.
    *   Генерация ответов на сообщения HR.

### MVP-4: «The Preparer» (Подготовитель)

Этот MVP вводит первый шаг к управлению процессом отклика, фокусируясь на подготовке и организации, а не на отправке.

*   **Целевая ценность:** Пользователь может собрать список вакансий, по которым он планирует откликнуться, и управлять этим списком.
*   **Обязательные функции:**
    *   **Очередь откликов:** Отдельная вкладка в Side Panel, где пользователь может добавлять вакансии из списка «Сохраненные». Каждая вакансия в очереди отображается с названием, компанией и кнопками «Просмотреть», «Удалить», «Отметить как отправлено».
    *   **Добавление в очередь:** Кнопка «Добавить в очередь» на каждой карточке вакансии в Side Panel и в поисковой выдаче.
    *   **Управление очередью:** При нажатии «Отметить как отправлено» статус вакансии в IndexedDB меняется на `applied`, и отправляется webhook-сообщение в n8n (если функция включена). Пользователь видит подтверждение и может отменить действие в течение 5 секунд.

*   **Что НЕ входит в MVP-4:**
    *   Автоматическая отправка откликов.
    *   Работа с разделом «Отклики» на сайте hh.ru.
    *   Классификация ответов HR.

### MVP-5: «The Manager» (Управляющий)

Этот MVP завершает первую итерацию, превращая расширение в полноценную систему управления карьерным процессом.

*   **Целевая ценность:** Пользователь получает полный контроль над всем своим циклом поиска работы: от первой оценки до последнего интервью.
*   **Обязательные функции:**
    *   **Работа с разделом «Отклики»:** Контент-скрипт читает данные из раздела «Мои отклики» на hh.ru и отображает их в Side Panel. Пользователь видит статус каждого отклика («Отправлено», «Ответили», «Пригласили на собеседование»).
    *   **Классификация ответов HR:** При появлении нового сообщения от HR в чате hh.ru расширение анализирует его текст и классифицирует: «Приглашение на собеседование», «Запрос на тестовое задание», «Отказ», «Другое». Классификация основана на rule-based шаблонах.
    *   **Генерация ответа HR:** Кнопка «Создать ответ» в чате HR. Расширение генерирует вежливый, профессиональный черновик ответа, который пользователь может скопировать и отправить.
    *   **Follow-up reminders:** Пользователь может установить напоминание о follow-up через 3 дня после отправки отклика. Напоминание отображается в popup-меню и отправляется в Telegram/n8n.

*   **Что НЕ входит в MVP-5:**
    *   Автоматическая отправка ответов HR.
    *   Интеграция с другими сайтами (LinkedIn, Indeed и т.д.). Это будет предметом отдельной, следующей фазы разработки.

Эта декомпозиция гарантирует, что каждый MVP является самостоятельным, ценным и безопасным продуктом. Она позволяет команде одного разработчика двигаться последовательно, получая постоянную обратную связь и минимизируя риски.

## Архитектура

Архитектура CareerSignal HH Copilot должна быть не просто технически реализуемой, а служить воплощением стратегических принципов безопасности, соответствия и пользовательского контроля. Она должна быть устойчивой к изменениям в веб-ландшафте и в то же время достаточно легковесной для быстрой разработки MVP. Ниже представлено детальное описание архитектурных решений, обоснованных на основе анализа предоставленных источников.

### Ядро: Manifest V3 и WXT Framework

Manifest V3 является неопровержимым стандартом де-факто для всех новых браузерных расширений в экосистеме Chromium [[233]]. Переход от устаревшего Manifest V2 к V3 был инициирован Google с целью повышения безопасности и производительности, и теперь он является обязательным требованием для публикации в Chrome Web Store [[46,48]]. Поэтому архитектура строится исключительно на Manifest V3. Для упрощения разработки и управления сложностью выбирается фреймворк WXT. WXT специально разработан для сборки расширений под Manifest V3 и предоставляет удобные абстракции для типичных задач: управление сервис-воркерами, контент-скриптами и popup-интерфейсами [[233]]. Он поддерживает TypeScript «из коробки» и отлично интегрируется с React или Solid.js, что позволяет создавать современные и поддерживаемые UI-компоненты [[229]].

### Frontend: React и TypeScript

Для реализации пользовательского интерфейса (popup, side panel, контент-скрипты) предпочтение отдается React. React является зрелой, хорошо документированной и широко используемой библиотекой, которая идеально подходит для создания сложных, интерактивных UI-компонентов, таких как Side Panel с фильтрацией и управлением карточками вакансий. Его декларативный подход упрощает управление состоянием и делает код более предсказуемым. Важнейшим архитектурным решением является использование TypeScript для всего frontend-кода. TypeScript добавляет статическую типизацию, которая является критически важной для проекта с такой сложной моделью данных, как `Job`, `Profile`, `CoverLetter`. Она предотвращает множество runtime-ошибок, упрощает рефакторинг и значительно повышает качество кода, что особенно ценно при работе в одиночку [[233]].

### Хранилище данных: IndexedDB и chrome.storage

Архитектура хранения данных строго следует принципу «local-first» и разделяет данные по их характеру и объему.

*   **Основная база данных (IndexedDB):** Для хранения всех структурированных данных о вакансиях, профилях и сопроводительных письмах используется IndexedDB [[197]]. Это асинхронная, транзакционная база данных, встроенная в браузер, которая предназначена для хранения больших объемов структурированных данных [[83]]. Она идеально подходит для нашей модели данных, так как позволяет создавать объектные хранилища (object stores) для каждой сущности (`jobs`, `profiles`, `cover_letters`) и выполнять сложные запросы с индексами (например, «найти все вакансии со статусом `saved`»). Важно отметить, что в Manifest V3 прямой доступ к IndexedDB из сервис-воркера ограничен, поэтому все операции с БД должны выполняться в контексте Content Script или Popup/Side Panel, а сервис-воркер должен взаимодействовать с ними через `runtime.sendMessage` [[78,79]]. Для упрощения работы с IndexedDB рекомендуется использовать библиотеку Dexie.js, которая предоставляет удобный, Promise-ориентированный API поверх низкоуровневого IndexedDB API [[77]].

*   **Настройки и конфигурация (chrome.storage.local):** Для хранения небольших объемов данных, таких как настройки пользователя (включена/выключена интеграция с n8n, выбранный профиль по умолчанию, настройки генерации писем), используется `chrome.storage.local` [[161]]. Это хранилище проще в использовании, чем IndexedDB, и идеально подходит для ключ-значение пар. Оно также обеспечивает синхронизацию настроек между устройствами пользователя, если он включен в синхронизацию Chrome.

### AI-стек: Secure Backend Proxy

Это наиболее критичный архитектурный компонент, от которого зависит безопасность всего продукта. Прямое использование API-ключей в браузере категорически запрещено. Вместо этого реализуется двухуровневая архитектура:

1.  **Frontend (Browser Extension):** Расширение собирает данные о вакансии и профиле, формирует структурированный запрос и отправляет его на свой backend-сервис через `fetch()` или `axios`. Запрос содержит только необходимые данные, а личная информация (email, телефон) предварительно маскируется.
2.  **Backend (Secure Proxy Service):** Это отдельный сервис, развернутый на облачной платформе (например, Vercel Serverless Functions или AWS Lambda). Он выполняет следующие функции:
    *   **Проверка и валидация:** Проверяет входящий запрос на корректность и соответствие ожидаемой схеме.
    *   **Маскировка данных:** Удаляет или заменяет все личные идентифицирующие данные (PII) из текста резюме и описания вакансии.
    *   **Интеграция с LLM:** Отправляет очищенный запрос к выбранному провайдеру ИИ (OpenAI, DeepSeek, OpenRouter) с использованием *своего* API-ключа, который никогда не покидает серверную среду.
    *   **Обработка ответа:** Получает ответ от LLM, форматирует его в структурированный JSON и возвращает клиенту.

Эта архитектура полностью устраняет риск утечки API-ключей и дает полный контроль над всеми входящими и исходящими данными. Она также позволяет легко добавлять новые провайдеры ИИ или реализовывать кэширование и лимитирование запросов.

### Связь с внешними системами: n8n Webhook

Интеграция с n8n реализована как опциональный и полностью контролируемый модуль. Архитектурно она представляет собой простой HTTP POST-запрос из расширения на URL, предоставленный пользователем. Чтобы обеспечить безопасность, реализуется следующий паттерн:

*   **Хранение URL:** URL webhook хранится в `chrome.storage.local` и доступен только для чтения/записи из popup-меню.
*   **Явное согласие:** Перед первым использованием пользователь должен ввести URL, подтвердить его ввод и просмотреть пример JSON-сообщения, которое будет отправлено. Сообщение должно быть максимально прозрачным и не содержать чувствительных данных.
*   **Минимизация данных:** В webhook-сообщении отправляются только те данные, которые необходимы для триггера в n8n: тип события (`job_viewed`, `application_sent`), ID вакансии, название вакансии и ссылка на нее. Полный текст описания или резюме в webhook не включаются.

### Безопасность и изоляция

Архитектура включает в себя несколько уровней изоляции для защиты данных пользователя:

*   **Контент-скрипты изолированы:** Контент-скрипты, встраиваемые в страницы hh.ru, работают в изолированном окружении и не имеют доступа к глобальному `window` объекту страницы. Это предотвращает конфликты с другим JavaScript-кодом на сайте.
*   **Сервис-воркер изолирован:** Сервис-воркер, управляющий фоновыми процессами, работает в своем собственном потоке и не имеет доступа к DOM, что предотвращает случайное или злонамеренное вмешательство в интерфейс сайта.
*   **Backend как барьер:** Backend-сервис выступает в роли «брандмауэра», который фильтрует и очищает все данные, проходящие между пользователем и внешними API.

Эта архитектура не является излишне сложной; она является необходимой и минимальной для достижения заявленных стратегических целей безопасности и соответствия.

## Data model

Модель данных — это фундамент, на котором строится вся логика продукта. Она должна быть простой, надежной, легко расширяемой и строго соответствовать принципам «local-first» и минимизации данных. Ниже представлена нормализованная и оптимизированная модель данных для CareerSignal HH Copilot, основанная на анализе исходных требований и технических ограничений.

### Core Entities

#### `Job` (Вакансия)
Это центральная сущность, представляющая одну вакансию с hh.ru. Все другие сущности связаны с ней.

```typescript
interface Job {
  id: string; // Уникальный ID вакансии, полученный из URL hh.ru (например, "123456789")
  sourceUrl: string; // Полный URL страницы вакансии на hh.ru
  title: string; // Название вакансии (например, "Senior Python Developer")
  company: string; // Название компании (например, "Tech Innovations LLC")
  salary: string | null; // Строка с зарплатой (например, "от 200 000 до 300 000 ₽"), null если не указана
  city: string | null; // Город (например, "Москва")
  typeOfWork: 'remote' | 'hybrid' | 'office' | null; // Тип занятости
  experience: string | null; // Строка с требуемым опытом (например, "от 3 лет")
  descriptionText: string; // Очищенный текст описания вакансии (без HTML-тегов)
  skills: string[]; // Массив ключевых навыков, извлеченных из вакансии
  aiAnalysis: string | null; // Текстовый отчет AI-анализа, сгенерированный backend-сервисом
  coverLetter: string | null; // Текст сопроводительного письма, сгенерированный backend-сервисом
  status: 'new' | 'viewed' | 'saved' | 'rejected_by_me' | 'letter_ready' | 'applied' | 'hr_replied' | 'interview' | 'test_task' | 'rejected_by_company' | 'offer' | 'blacklist';
  profileId: string | null; // ID профиля, использованного для анализа и генерации письма
  createdAt: number; // Timestamp создания записи (Date.now())
  updatedAt: number; // Timestamp последнего обновления
}
```

**Ключевые особенности:**
*   **`id`** используется как первичный ключ в IndexedDB и для всех внешних ссылок.
*   **`descriptionText`**, а не `descriptionHtml`, — это стратегическое решение для минимизации данных и обеспечения безопасности. Полный HTML не сохраняется, что соответствует требованиям и снижает риски [[41]].
*   **`status`** — это единый, расширяемый enum, который охватывает весь жизненный цикл вакансии, от первого просмотра до получения предложения.

#### `Profile` (Профиль кандидата)
Это сущность, представляющая «персону» пользователя, его резюме и ключевые компетенции.

```typescript
interface Profile {
  id: string; // Уникальный ID профиля (UUID)
  name: string; // Имя профиля (например, "AI Engineer / LLM / RAG")
  resumeText: string; // Очищенный текст резюме (без контактных данных)
  skills: string[]; // Массив ключевых навыков пользователя
  createdAt: number;
  updatedAt: number;
}
```

**Ключевые особенности:**
*   **`resumeText`** должен быть очищен от PII (личной идентифицирующей информации) перед сохранением. Это делается на уровне frontend-логики перед отправкой на backend для AI-анализа.
*   Профили хранятся в отдельном объектном хранилище `profiles` в IndexedDB.

#### `CoverLetter` (Сопроводительное письмо)
Хотя текст письма хранится в поле `Job.coverLetter`, для будущей гибкости и возможности управления несколькими вариантами писем, рекомендуется вынести его в отдельную сущность.

```typescript
interface CoverLetter {
  id: string; // UUID
  jobId: string; // Ссылка на Job.id
  profileId: string; // Ссылка на Profile.id
  content: string; // Текст письма
  template: 'hh_standard' | 'telegram' | 'en_version'; // Использованный шаблон
  lengthLimit: number | null; // Ограничение длины (500, 1000), null если нет
  createdAt: number;
}
```

**Ключевые особенности:**
*   Это позволяет пользователю хранить несколько вариантов писем для одной вакансии и выбирать наиболее подходящий.
*   Связь `jobId` и `profileId` обеспечивает полную прозрачность, кто и для чего сгенерировал письмо.

### Supporting Entities

#### `Application` (Отклик)
Эта сущность фокусируется на самом факте отклика и его метаданных.

```typescript
interface Application {
  id: string; // UUID
  jobId: string; // Ссылка на Job.id
  dateApplied: number; // Timestamp отправки отклика
  status: 'sent' | 'replied' | 'interview_scheduled' | 'test_task_assigned' | 'offer_received' | 'rejected';
  hrResponse: string | null; // Текст первого ответа от HR (для MVP-5)
  followUpDate: number | null; // Timestamp для напоминания о follow-up
}
```

**Ключевые особенности:**
*   `Application` не хранит полный текст отклика, так как он уже сохранен в `Job.coverLetter`. Она хранит только факт и статус взаимодействия.

#### `Event` (Событие)
Для интеграции с n8n и внутренних уведомлений используется сущность `Event`.

```typescript
interface Event {
  id: string; // UUID
  type: 'job_viewed' | 'job_saved' | 'application_sent' | 'hr_replied' | 'strong_vacancy_found';
  jobId: string | null;
  payload: Record<string, any>; // Минимальный, структурированный payload для webhook
  timestamp: number;
}
```

**Ключевые особенности:**
*   `payload` должен быть строго ограничен и не содержать чувствительных данных. Например, для `job_viewed`: `{ jobId: "123456789", title: "Senior Python Developer", url: "https://hh.ru/vacancy/123456789" }`.

### Database Schema in IndexedDB

Для реализации этой модели в IndexedDB будет создано несколько объектных хранилищ (object stores):

*   **`jobs`:** Хранит все объекты `Job`. Первичный ключ — `id`. Индексы: `status`, `createdAt`, `profileId`.
*   **`profiles`:** Хранит все объекты `Profile`. Первичный ключ — `id`.
*   **`cover_letters`:** Хранит все объекты `CoverLetter`. Первичный ключ — `id`. Индексы: `jobId`, `profileId`.
*   **`applications`:** Хранит все объекты `Application`. Первичный ключ — `id`. Индексы: `jobId`, `dateApplied`.
*   **`events`:** Хранит все объекты `Event`. Первичный ключ — `id`. Индекс: `timestamp`.

Эта схема обеспечивает высокую производительность при выполнении типичных запросов (например, «получить все вакансии со статусом `saved`») и при этом остается простой и понятной для поддержки одним разработчиком.

## Scoring model

Система оценки вакансий является ключевым элементом пользовательского опыта, так как она предоставляет пользователю быструю, визуальную оценку релевантности. В рамках исправленной стратегии, которая делает ставку на безопасность и прозрачность, гибридная модель (rule-based + AI) переосмысливается. AI-анализ не используется для генерации числового балла, а служит для создания качественного, понятного человеку текстового отчета. Rule-based scoring, в свою очередь, должен быть простым, надежным и легко объяснимым. Ниже представлена уточненная и оптимизированная модель.

### Rule-Based Scoring Model for HH-Вакансий

Модель основана на концепции «взвешенного совпадения» и «штрафов за несоответствие». Она не пытается имитировать сложные ML-алгоритмы, а фокусируется на том, что можно точно и надежно извлечь из структурированных данных вакансии. Балл рассчитывается по формуле:

$$\text{Score} = \frac{\text{Total Points}}{\text{Max Possible Points}} \times 100$$

**Базовые баллы (за совпадение):**

| Критерий | Баллы | Объяснение |
| :--- | :--- | :--- |
| **Совпадение навыков** | $+10$ | За каждый навык из `Profile.skills`, который найден в `Job.skills`. Максимум 50 баллов (5 совпадений). |
| **Совпадение в описании** | $+5$ | За каждое вхождение любого навыка из `Profile.skills` в `Job.descriptionText` (с учетом словоформ). Максимум 25 баллов (5 вхождений). |
| **Совпадение типа занятости** | $+3$ | Если `Profile.preferredTypeOfWork` (например, `'remote'`) совпадает с `Job.typeOfWork`. |
| **Совпадение города** | $+2$ | Если `Profile.preferredCity` совпадает с `Job.city`. |

**Штрафы (за несоответствие):**

| Критерий | Штраф | Объяснение |
| :--- | :--- | :--- |
| **Нехватка ключевых навыков** | $-10$ | Если в `Job.skills` есть хотя бы один навык, который указан как «обязательный» в `Profile.requiredSkills`, но отсутствует в `Profile.skills`. |
| **Несоответствие опыта** | $-15$ | Если `Job.experience` требует опыт, который превышает `Profile.yearsOfExperience`. Например, вакансия требует «от 5 лет», а профиль указывает «3 года». |
| **Несоответствие зарплаты** | $-10$ | Если `Job.salary` ниже `Profile.minSalaryExpectation`. |

**Максимально возможные баллы:** $50 + 25 + 3 + 2 = 80$. Таким образом, максимальный балл в системе составляет 80, что нормализуется до 100-балльной шкалы.

**Классификация по баллу:**
*   **Сильная вакансия:** $Score \geq 85$ (что соответствует 71.25 из 80 возможных)
*   **Средняя вакансия:** $60 \leq Score < 85$
*   **Слабая вакансия:** $Score < 60$

**Пример расчета:**
*   Профиль: `skills = ["Python", "Django", "PostgreSQL"]`, `preferredTypeOfWork = "remote"`, `minSalaryExpectation = 200000`.
*   Вакансия: `skills = ["Python", "Django", "AWS"]`, `descriptionText = "Требуется Python и Django..."`, `typeOfWork = "remote"`, `salary = "от 250 000 ₽"`.
*   Расчет: $+10$ (Python) $+10$ (Django) $+5$ (Python в описании) $+5$ (Django в описании) $+3$ (remote) $= 33$ балла из 80. Нормализованный балл: $\frac{33}{80} \times 100 = 41.25$. Классификация: «Слабая вакансия».

Эта модель проста для понимания пользователя: он видит, что балл низкий из-за отсутствия навыка «AWS» в его профиле. Это создает доверие и прозрачность, в отличие от «черного ящика» ИИ, который просто выдает число без объяснения.

### AI-Powered Analysis (Non-Scoring)

AI используется не для вычисления балла, а для генерации текстового отчета, который дополняет rule-based анализ. Этот отчет должен быть структурированным и полезным:

*   **Сильные стороны:** «Вы обладаете всеми ключевыми навыками, указанными в вакансии: Python и Django. Ваш опыт в разработке веб-приложений напрямую соответствует описанию задач».
*   **Чего не хватает в резюме:** «В вакансии указан навык AWS, которого нет в вашем профиле. Рекомендуем добавить его, если у вас есть опыт».
*   **Почему стоит откликнуться?:** «Вакансия предлагает удаленную работу и зарплату выше вашего ожидания, что делает ее очень привлекательной».

Такой подход сочетает в себе надежность и скорость rule-based анализа с глубиной и персонализацией ИИ, не создавая при этом рисков и не скрывая логику от пользователя.

## UX/UI структура

UX/UI структура CareerSignal HH Copilot — это не просто набор интерфейсов, а продуманная экосистема, которая интегрируется в пользовательский рабочий процесс на hh.ru, не нарушая его и не создавая когнитивной нагрузки. Она строится вокруг принципа «контекстуального присутствия», где информация и инструменты появляются там, где они нужны, и исчезают, когда не нужны. Ниже представлена детальная архитектура UX/UI.

### 1. Content Script: Интеллектуальный слой на странице hh.ru

Content Script — это «невидимый помощник», который работает на самой странице вакансии или поисковой выдачи. Он не перекрывает контент, а добавляет к нему дополнительные, информативные элементы.

*   **На странице вакансии:**
    *   **Панель действий:** Горизонтальная панель с кнопками, расположенная под заголовком вакансии. Кнопки: «Просмотрено» (меняет статус на `viewed`), «Сохранить» (меняет статус на `saved`), «AI-анализ» (открывает всплывающий отчет в боковой панели), «Сгенерировать письмо» (открывает форму генерации в боковой панели).
    *   **Бейджи:** Маленький цветной бейдж (например, зеленый) в правом верхнем углу заголовка вакансии, отображающий rule-based score и классификацию («Сильная»).

*   **На странице поисковой выдачи:**
    *   **Бейджи на карточках:** На каждой карточке вакансии в результатах поиска добавляется маленький бейдж в правом верхнем углу. Цвет и текст бейджа зависят от статуса и score: зеленый «Сильная» для `score >= 85`, желтый «Средняя» для `60 <= score < 85`, серый «Слабая» для `score < 60`.
    *   **Подсветка карточек:** Карточки вакансий подсвечиваются в соответствии с их статусом в IndexedDB: `new` — голубой фон, `viewed` — светло-серый, `saved` — светло-зеленый, `rejected_by_me` — светло-красный. Это создает визуальную карту истории пользователя.

### 2. Side Panel: Центр управления

Side Panel — это главное рабочее пространство расширения, встраиваемое в правую часть любого сайта hh.ru. Он является «центральным офисом» пользователя.

*   **Структура:**
    *   **Заголовок:** «CareerSignal: Мои вакансии» с кнопкой «Скрыть/Показать».
    *   **Фильтры и поиск:** Верхняя панель с фильтрами по статусу (`all`, `saved`, `new`, `applied`) и строкой поиска по названию и компании.
    *   **Список вакансий:** Вертикальный список карточек. Каждая карточка содержит:
        *   Заголовок и компания (жирным).
        *   Бейдж с score и классификацией.
        *   Статусный индикатор (иконка глаза, сердца, галочки).
        *   Краткое описание (первые 50 символов из `descriptionText`).
        *   Кнопки действий: «AI-анализ», «Сгенерировать письмо», «Добавить в очередь», «Удалить».
    *   **Вкладки:** Внизу панели — вкладки: «Все вакансии», «Очередь», «Профили», «Настройки».

### 3. Popup: Краткий обзор и управление

Popup — это компактное меню, которое открывается при клике на иконку расширения в панели браузера. Его цель — предоставить быстрый доступ к ключевой информации и настройкам.

*   **Структура:**
    *   **Статистика:** Числа: «Новых: X», «Сохранено: Y», «Отправлено: Z».
    *   **Быстрые ссылки:** Кнопки «Открыть Side Panel», «Добавить профиль», «Настройки n8n».
    *   **Недавние события:** Список последних 3 событий из `events` (например, «Сильная вакансия найдена», «Отклик отправлен»).
    *   **Индикаторы:** Индикатор подключения к backend-сервису и статус интеграции с n8n.

### 4. Settings Dashboard: Управление настройками

Dashboard — это полноценная страница, доступная по ссылке в popup. Это место для детальной настройки.

*   **Разделы:**
    *   **Профили:** Интерфейс для создания, редактирования и удаления профилей (`Profile`). Включает форму с полями `name`, `resumeText`, `skills`.
    *   **Настройки AI:** Выбор провайдера ИИ (OpenAI, DeepSeek, OpenRouter), настройка параметров генерации (температура, длина).
    *   **Интеграции:** Форма для ввода URL n8n webhook с кнопкой «Проверить подключение» и примером JSON-сообщения.
    *   **Экспорт:** Кнопки «Экспорт всех вакансий (CSV)», «Экспорт всех вакансий (JSON)», «Экспорт только сохраненных».
    *   **Безопасность:** Кнопка «Очистить все данные» (удаляет все из IndexedDB и `chrome.storage`).

Эта структура обеспечивает идеальный баланс между вовлеченностью и невидимостью. Пользователь получает всю необходимую информацию и инструменты, не покидая привычного интерфейса hh.ru, и при этом не ощущает, что расширение «навязывается» ему. Каждый компонент имеет четкую, непересекающуюся функцию, что делает продукт интуитивно понятным и простым в освоении.

## AI prompts

Промпты для генерации ИИ-анализа и сопроводительных писем являются «душой» продукта. Они определяют качество и полезность выводов, которые пользователь получает от расширения. В рамках исправленной стратегии, которая делает ставку на прозрачность и безопасность, промпты должны быть тщательно продуманы, чтобы гарантировать, что генерируемый текст будет полезным, этичным и не будет содержать вредоносных или неуместных рекомендаций. Ниже представлены оптимизированные шаблоны промптов, которые будут использоваться в backend-сервисе.

### 1. Prompt для AI-анализа вакансии

Этот промпт используется для генерации текстового отчета, который отображается в Side Panel. Он не должен генерировать числовой балл, а должен фокусироваться на качественном, структурированном анализе.

```text
Ты — эксперт по подбору персонала и карьерному консультированию. Твоя задача — помочь кандидату принять взвешенное решение о том, стоит ли откликаться на вакансию. Ты получишь описание вакансии и краткое резюме кандидата. Проанализируй их и предоставь ответ в строго указанном формате, без каких-либо дополнительных комментариев.

**Входные данные:**
- Описание вакансии: {job_description}
- Навыки вакансии: {job_skills}
- Требуемый опыт: {job_experience}
- Зарплата: {job_salary}
- Тип занятости: {job_type_of_work}
- Город: {job_city}
- Резюме кандидата: {profile_resume_text}
- Навыки кандидата: {profile_skills}
- Предпочтительный тип занятости кандидата: {profile_preferred_type_of_work}
- Предпочтительный город кандидата: {profile_preferred_city}
- Минимальная ожидаемая зарплата кандидата: {profile_min_salary_expectation}

**Формат ответа (строго соблюдай!):**
### Сильные стороны
{Текст, перечисляющий 2-3 ключевых совпадения, которые делают вакансию привлекательной для кандидата. Сосредоточься на навыках, опыте и условиях.}

### Чего не хватает в резюме
{Текст, перечисляющий 1-2 конкретных навыка или требований из вакансии, которых нет в резюме кандидата. Дай краткую рекомендацию по их добавлению.}

### Почему стоит откликнуться?
{Краткий (до 3 предложений) вывод, объясняющий общую совместимость. Упомяни ключевые факторы: зарплата, тип занятости, город и общую релевантность.}

### Рекомендация
{Одно из трех: "Откликаться", "Подумать", "Пропустить". Обоснуй свой выбор в одном предложении.}
```

**Ключевые особенности:**
*   **Строгий формат:** Гарантирует, что ответ будет структурированным и легко парсится frontend-кодом для отображения в Side Panel.
*   **Фокус на полезности:** Промпт явно указывает ИИ, что его задача — помочь в принятии решения, а не просто описать факты.
*   **Безопасность:** Промпт не содержит инструкций, которые могли бы привести к генерации вредоносного или неэтичного контента. Он фокусируется на анализе и рекомендациях.

### 2. Prompt для генерации сопроводительного письма

Этот промпт используется для создания персонализированного черновика письма. Он должен быть гибким, чтобы учитывать различные шаблоны и настройки пользователя.

```text
Ты — профессиональный копирайтер, специализирующийся на составлении сопроводительных писем для IT-специалистов. Твоя задача — создать краткое, вежливое и убедительное письмо, которое подчеркивает сильные стороны кандидата и его соответствие вакансии.

**Входные данные:**
- Описание вакансии: {job_description}
- Навыки вакансии: {job_skills}
- Требуемый опыт: {job_experience}
- Название вакансии: {job_title}
- Название компании: {job_company}
- Резюме кандидата: {profile_resume_text}
- Навыки кандидата: {profile_skills}
- Шаблон письма: {template} (возможные значения: "hh_standard", "telegram", "en_version")
- Ограничение длины: {length_limit} (например, 500)

**Инструкции:**
- Письмо должно быть написано от первого лица.
- Начни с приветствия ("Уважаемая команда [Компания]!" для hh_standard/en_version, "Привет!" для telegram).
- В первом абзаце кратко представь кандидата и укажи, на какую вакансию он откликается.
- Во втором абзаце перечисли 2-3 ключевых совпадения между навыками кандидата и требованиями вакансии. Используй конкретные примеры из резюме.
- В третьем абзаце вырази энтузиазм по поводу компании и вакансии.
- Заверши письмо вежливым прощанием ("С уважением," для hh_standard/en_version, "Спасибо!" для telegram).
- Не используй смайлики, если шаблон не telegram.
- Не используй markdown-разметку.
- Строго соблюдай ограничение длины. Если письмо длиннее, сократи его, сохранив ключевые моменты.
- Не упоминай зарплату или другие финансовые вопросы.
- Не используй фразы "Я считаю", "Я думаю". Пиши утвердительно.

**Формат ответа:**
{Только текст письма, без каких-либо дополнительных комментариев или заголовков.}
```

**Ключевые особенности:**
*   **Гибкость:** Промпт учитывает шаблон и ограничение длины, что позволяет генерировать письма для разных целей.
*   **Профессионализм:** Четкие инструкции по стилю и структуре гарантируют, что письмо будет выглядеть профессионально и соответствовать ожиданиям рекрутеров.
*   **Безопасность:** Явный запрет на упоминание зарплаты и использование неуместных фраз минимизирует риски генерации неподходящего контента.

Эти промпты являются не статичными шаблонами, а живыми компонентами, которые будут постоянно улучшаться на основе обратной связи пользователей и анализа качества генерируемых ответов.

## Security/privacy

Модель безопасности и приватности CareerSignal HH Copilot является не набором технических мер, а фундаментальным принципом, пронизывающим всю архитектуру и дизайн продукта. Она основана на трех китах: **минимизации данных**, **максимальной прозрачности** и **полной изоляции секретов**. Ниже представлено детальное описание всех компонентов этой модели.

### Permissions Model

Расширение запрашивает только те разрешения, которые абсолютно необходимы для выполнения своих функций. Это является критически важным требованием для доверия пользователей и для прохождения модерации в Chrome Web Store.

*   **Обязательные разрешения:**
    *   `activeTab`: Необходимо для чтения данных со страницы вакансии, на которой пользователь находится в данный момент. Это разрешение является самым безопасным, так как оно дает доступ только к текущей вкладке и не требует указания конкретных доменов.
    *   `storage`: Необходимо для хранения настроек пользователя в `chrome.storage.local`.

*   **Запрещенные разрешения:**
    *   `tabs`: Не требуется, так как расширение не читает или не управляет другими вкладками.
    *   `declarativeNetRequest`: Не требуется для MVP, так как расширение не блокирует рекламу или не изменяет сетевые запросы.
    *   `host_permissions` для `*://*.hh.ru/*`: Это разрешение было бы слишком широким и рискованным. Вместо него используется `activeTab`, что обеспечивает безопасность и достаточный функционал.

### Local Storage and Data Minimization

Принцип «local-first» реализуется через строгую политику минимизации данных.

*   **Что хранится локально:**
    *   **`Job` объекты:** Только структурированные поля (`title`, `company`, `salary`, `city`, `typeOfWork`, `experience`, `skills`) и очищенный текст описания (`descriptionText`). Полный HTML страницы **не сохраняется** [[41]].
    *   **`Profile` объекты:** Только имя профиля, очищенный текст резюме и массив навыков. Контактные данные (email, телефон, имя, адрес) **маскируются или удаляются** перед сохранением.
    *   **Настройки:** Только те параметры, которые пользователь явно настраивает (выбранный профиль, настройки генерации писем, URL n8n webhook).

*   **Что НЕ хранится локально:**
    *   Полный HTML-код страниц вакансий или поисковой выдачи.
    *   Скриншоты или медиафайлы.
    *   История посещений или любые данные, не относящиеся напрямую к вакансиям и профилям.

Эта политика гарантирует, что даже в случае компрометации расширения злоумышленник получит доступ только к тем данным, которые пользователь сам решил сохранить в рамках процесса поиска работы.

### AI Data Handling and Anonymization

Обработка данных для ИИ-запросов — это наиболее чувствительная зона, и здесь применяются самые строгие меры.

*   **Обязательная маскировка:** Перед тем как данные о вакансии и профиле будут отправлены на backend-сервис, frontend выполняет их предварительную обработку:
    *   Из `Profile.resumeText` удаляются все email-адреса, номера телефонов и полные имена. Они заменяются на шаблоны (например, `[EMAIL]`, `[PHONE]`, `[FULL_NAME]`).
    *   Из `Job.descriptionText` удаляются любые упоминания конкретных лиц или внутренних систем компании, если они не являются ключевыми навыками.
*   **Backend-валидация:** Backend-сервис повторно проверяет входящий запрос и отвергает его, если в нем обнаружены неочищенные PII-данные.
*   **No Logging Policy:** Backend-сервис не ведет логов входящих запросов или исходящих ответов. Это гарантирует, что данные пользователя не остаются на сервере после обработки.

### n8n Webhook Safety

Интеграция с n8n реализована как полностью опциональный и контролируемый модуль.

*   **Явное согласие:** URL webhook хранится только в `chrome.storage.local` и доступен для редактирования только в настройках. Пользователь должен ввести URL, подтвердить его и просмотреть пример JSON-сообщения, которое будет отправлено.
*   **Минимизация payload:** В webhook-сообщении отправляются только те данные, которые необходимы для триггера: тип события, ID вакансии, название вакансии и ссылка на нее. Никаких описаний, резюме или личных данных.
*   **Отключение по умолчанию:** Функция интеграции с n8n отключена по умолчанию и активируется только после явного согласия пользователя.

### Encryption and Isolation

Для MVP-1 и MVP-2 шифрование локальных данных в IndexedDB не является обязательным требованием, так как IndexedDB работает в изолированном контексте расширения, и доступ к нему имеют только скрипты этого расширения. Однако для будущих версий, особенно когда в расширении появятся более чувствительные данные, будет реализована опциональная функция шифрования с использованием Web Crypto API.

Эта комплексная модель безопасности и приватности не является избыточной. Она является необходимым условием для создания продукта, которому пользователи будут доверять, и который сможет успешно пройти все этапы модерации и соответствовать самым строгим требованиям.

### Detailed Mitigation Strategies

**1. Mitigation for hh.ru Blocking Risk:**  
Это главный риск, и его смягчение — приоритет №1. Помимо отказа от подстановки текста, необходимо внедрить «человеческое поведение». Сервис-воркер будет генерировать случайные, но небольшие задержки (от 1 до 5 секунд) между любыми двумя последовательными действиями пользователя, инициированными расширением (например, между нажатием «Сохранить» и сохранением данных в IndexedDB). Это имитирует естественное поведение человека и снижает вероятность срабатывания детекторов ботов, которые ищут «идеально» быстрые и последовательные действия [[31]].

**2. Mitigation for ToS Violation Risk:**  
Для обеспечения соответствия необходимо создать внутренний «чек-лист соответствия», который будет применяться к каждой новой функции перед ее реализацией. Чек-лист будет включать вопросы: «Эта функция читает данные, которые видит пользователь?», «Эта функция отправляет какие-либо данные на сервер hh.ru?», «Эта функция изменяет DOM-элементы на странице hh.ru?». Если на любой вопрос получен ответ «Нет», функция не может быть реализована в MVP.

**3. Mitigation for API Key Leakage Risk:**  
Backend-сервис должен быть развернут с использованием «serverless functions», где API-ключи хранятся в защищенных переменных окружения (environment variables), недоступных для чтения извне. Кроме того, сервис должен вести журнал всех входящих запросов (без сохранения самих данных), чтобы можно было отследить аномальную активность и вовремя заблокировать злоумышленника.

**4. Mitigation for PII Leakage Risk:**  
Для маскировки PII будет использована библиотека `pii-masker`, которая умеет надежно находить и заменять email, телефоны и имена в тексте. Кроме того, в настройках расширения будет добавлена опция «Полная анонимизация», которая будет удалять из резюме не только контакты, но и любые упоминания компаний и проектов, оставляя только навыки и опыт.

**5. Mitigation for AI Expectation Risk:**  
В интерфейсе будет добавлено небольшое пояснение рядом с кнопкой «AI-анализ»: «Это ИИ-анализ, основанный на ваших данных. Результаты не являются гарантией, а служат для помощи в принятии решения. Всегда проверяйте и редактируйте сгенерированный текст». Это устанавливает правильные ожидания и снижает риск разочарования.

Эти меры не являются пассивными «планами Б». Они являются активными, интегрированными компонентами разработки, которые будут внедрены с самого первого дня работы над MVP-1.

## Roadmap

Roadmap для одного разработчика — это не просто график, а стратегический план, который балансирует между скоростью выхода на рынок, качеством продукта и управлением рисками. Он построен на принципе «построить, измерить, научиться», где каждый этап MVP является законченным продуктом, который можно выпустить и получить обратную связь. Ниже представлен подробный, недельный roadmap, рассчитанный на 12 недель для MVP-1–MVP-4, с последующим планированием MVP-5.

### Phase 1: Foundation & MVP-1 «The Scout» (Weeks 1–4)

*   **Week 1: Setup & Architecture**
    *   Инициализация проекта с WXT и TypeScript.
    *   Настройка CI/CD (GitHub Actions) для автоматической сборки.
    *   Создание базовой структуры Manifest V3 и настройка `content_scripts`.
    *   Настройка `chrome.storage.local` для настроек.
*   **Week 2: Data Layer & Extraction**
    *   Интеграция Dexie.js и создание объектных хранилищ `jobs` и `profiles`.
    *   Реализация контент-скрипта для извлечения данных со страницы вакансии hh.ru.
    *   Тестирование извлечения на 10 различных вакансиях.
*   **Week 3: Core Logic & UI**
    *   Реализация системы статусов (`new`, `viewed`, `saved`).
    *   Создание popup-меню с базовой статистикой.
    *   Разработка и интеграция rule-based scoring модели.
    *   Создание бейджа с score на странице вакансии.
*   **Week 4: AI Integration & Export**
    *   Развертывание backend-сервиса (Vercel Functions).
    *   Интеграция frontend с backend для AI-анализа.
    *   Реализация функции экспорта вакансий в CSV/JSON.
    *   Тестирование MVP-1 вручную на сайте hh.ru.

### Phase 2: Expansion & MVP-2 «The Organizer» (Weeks 5–8)

*   **Week 5: Side Panel Development**
    *   Создание базового интерфейса Side Panel с использованием React.
    *   Реализация загрузки и отображения всех вакансий из IndexedDB.
*   **Week 6: UI Enhancements & Profiles**
    *   Добавление фильтров и поиска в Side Panel.
    *   Реализация менеджера профилей (`Profile`) в popup-меню.
    *   Интеграция профилей с AI-анализом.
*   **Week 7: Search Page Integration**
    *   Разработка контент-скрипта для поисковой выдачи hh.ru.
    *   Реализация бейджей и подсветки карточек вакансий.
*   **Week 8: Testing & Polish**
    *   Комплексное ручное тестирование на всех типах страниц hh.ru.
    *   Исправление критических багов и оптимизация производительности.
    *   Подготовка MVP-2 к внутреннему релизу.

### Phase 3: Automation & MVP-3/4 «The Assistant» & «The Preparer» (Weeks 9–12)

*   **Week 9: Cover Letter Generation**
    *   Разработка и интеграция промпта для генерации писем.
    *   Реализация интерфейса генерации в Side Panel.
    *   Добавление шаблонов и настроек длины.
*   **Week 10: Queue Management**
    *   Создание вкладки «Очередь» в Side Panel.
    *   Реализация логики добавления и удаления вакансий из очереди.
    *   Интеграция с n8n webhook для события «отправлено».
*   **Week 11: Advanced Features**
    *   Реализация рекомендации резюме на основе анализа.
    *   Добавление функции «Копировать письмо» с анимацией подтверждения.
*   **Week 12: Release & Feedback**
    *   Подготовка MVP-3 и MVP-4 к публикации в Chrome Web Store (через внутренний канал).
    *   Запуск программы бета-тестирования с 10–20 пользователями.
    *   Сбор и анализ первой обратной связи.

### Phase 4: Future & MVP-5 «The Manager» (Weeks 13+)

*   **Week 13–14:** Анализ обратной связи от бета-тестеров и приоритизация задач для MVP-5.
*   **Week 15–16:** Разработка контент-скрипта для раздела «Мои отклики».
*   **Week 17–18:** Реализация классификации ответов HR и генерации ответов.
*   **Week 19–20:** Добавление системы напоминаний (follow-up) и интеграции с Telegram.
*   **Week 21+:** Подготовка к публикации MVP-5 и начало работы над поддержкой других платформ (LinkedIn, Indeed).

Этот roadmap является гибким. Каждая неделя заканчивается «точкой принятия решения», где разработчик оценивает прогресс и при необходимости корректирует план. Главная цель — не просто завершить 12 недель, а выпустить четыре работающих, ценных продукта, каждый из которых приносит реальную пользу пользователю.

## Список конкретных улучшений, которые стоит добавить в ТЗ

На основе проведенного критического аудита и анализа, вот конкретный, приоритизированный список улучшений, которые должны быть немедленно внесены в техническое задание (ТЗ) для CareerSignal HH Copilot. Эти улучшения не являются опциональными «хотелками», а являются обязательными корректировками, необходимыми для обеспечения жизнеспособности и соответствия продукта.

1.  **Заменить пункт «полуавтоматический отклик» на «ручную подготовку черновика»:** В ТЗ должно быть четко указано, что функция «подготовка письма» (MVP-3) предоставляет пользователю готовый текст для копирования в буфер обмена. Любая форма автоматической подстановки текста в элементы `<form>` или `<textarea>` на странице hh.ru должна быть **полностью исключена** из всех MVP-версий. Вместо этого должна быть добавлена кнопка «Копировать письмо» с визуальной обратной связью (например, анимация и надпись «Скопировано!»).

2.  **Добавить обязательный пункт «Backend-сервис для ИИ»:** В разделе «AI: BYOK» ТЗ должно быть добавлено требование: «Все ИИ-запросы должны выполняться через промежуточный backend-сервис. Хранение и использование API-ключей в клиентском коде (JavaScript/TypeScript) строго запрещено». Должны быть указаны минимальные требования к этому сервису: поддержка маскировки PII, валидации запросов и отсутствие логирования.

3.  **Уточнить политику хранения данных:** В ТЗ должно быть четко прописано: «Полный HTML-код страниц вакансий и поисковой выдачи **не сохраняется** в IndexedDB. Сохраняется только очищенный текст описания (`descriptionText`) и структурированные поля (название, компания, зарплата и т.д.)». Это напрямую соответствует требованиям условий использования hh.ru [[41]].

4.  **Добавить пункт «Явное согласие на ИИ-запросы»:** В ТЗ должно быть добавлено требование к UX: «Перед каждым ИИ-запросом пользователь должен видеть диалоговое окно, в котором отображается точный список данных, которые будут отправлены на сервер (например, `job_title`, `job_skills`, `profile_skills`). Пользователь должен дать явное согласие (кнопка «Отправить») или отменить запрос (кнопка «Отмена»)».

5.  **Уточнить модель интеграции с n8n:** В ТЗ должно быть добавлено требование: «Интеграция с n8n является полностью опциональным модулем. По умолчанию она отключена. Для ее активации пользователь должен вручную ввести URL webhook, подтвердить его дважды и просмотреть пример JSON-сообщения, которое будет отправлено. В webhook-сообщении не должны присутствовать никакие персональные данные пользователя или полные тексты резюме/описаний».

6.  **Добавить пункт «Регулярный аудит соответствия»:** В ТЗ должно быть добавлено требование к процессу разработки: «Перед началом реализации каждой новой функции команда должна провести аудит на соответствие Условиям использования hh.ru. Аудит должен быть задокументирован и включать ответы на вопросы: 1) Эта функция читает только то, что видит пользователь? 2) Эта функция отправляет какие-либо данные на сервер hh.ru? 3) Эта функция изменяет DOM-элементы на странице hh.ru?». Ответ «Нет» на любой вопрос делает функцию неприемлемой для MVP.

7.  **Уточнить модель данных для `Profile`:** В ТЗ должно быть добавлено требование: «Поле `Profile.resumeText` должно быть очищено от всех личных идентифицирующих данных (email, телефон, имя, адрес) перед сохранением в `chrome.storage.local`. Должна быть реализована функция маскировки, которая заменяет эти данные на шаблоны (например, `[EMAIL]`, `[PHONE]`)».

Эти семь пунктов являются критическими улучшениями, которые трансформируют ТЗ из потенциально рискованного документа в надежный и этичный план разработки. Их внедрение является обязательным условием для начала работы над MVP-1.

Executive Summary
Концепция CareerSignal HH Copilot — одна из самых продуманных начальных спецификаций «не‑спам»‑расширения для рынка труда, которые я видел.
Она правильно ставит во главу угла безопасность пользователя, приватность, контроль над автоматизацией и архитектурную чистоту.
Тем не менее есть несколько противоречий между «быстрым MVP» и «глубокой автоматизацией», а также недооценённые риски, связанные с DOM‑мутациями, антибот‑защитой и объёмом первой версии.

Ниже — критический аудит, исправленная продуктовая стратегия, архитектура, модели данных, AI‑промпты, планы безопасности и роадмап, оптимизированный под одного разработчика.

Что в концепции сильное
User‑controlled copilot, а не бот. Никакой автоотправки без подтверждения.

Local‑first, BYOK для AI, минимум данных наружу — отличный приватный дефолт.

Manifest V3 + WXT + TypeScript + Dexie — современный, расширяемый стек.

Гибридный скоринг: правило + AI по требованию — экономит деньги и повышает скорость.

Профили кандидата и шаблоны писем — сразу закладывается персонализация, а не универсальный спам.

MVP строго на HH.ru, но архитектура рассчитана на будущие сайты — правильный баланс.

Статусная модель почти полностью покрывает реальный флоу кандидата.

Отказ от обязательного бэкенда — снижает порог входа и риск утечек.

Правильные приоритеты: не получить бан → качество откликов → скорость MVP.

Что в концепции опасно или спорно
Слишком много MVP‑1.
Анализ, сохранение, AI‑генерация, экспорт, webhook, 10 статусов, rule‑based score, профили, шаблоны — это всё сразу в первой версии создаст высокую когнитивную нагрузку и длительный цикл разработки. Риск: разработчик утонет в фичах, а базовый парсинг HH ещё нестабилен.

DOM‑мутации на странице поиска HH (MVP‑2) без фазы наблюдателя.
HH.ru активно обновляет вёрстку, A/B‑тестирует, использует динамическую подгрузку. Без «тихого» периода сбора реальных селекторов и тестирования на разных аккаунтах можно получить неработающие бейджи или (хуже) срабатывание антибот‑скриптов.

«Очередь откликов» в MVP‑3 и MVP‑4 может быть воспринята HH как автоматизированная активность.
Даже с подтверждением, последовательная подстановка текста и клик «Отправить» с интервалами — паттерн, который системы защиты могут задетектировать. Надо проектировать с «человеческими» задержками и без инжекта в защищённые элементы.

Нет чёткого разделения между «извлечением данных» и «модификацией страницы».
Content script, который одновременно парсит вакансию и встраивает UI на страницу, может конфликтовать с CSP, блокировщиками рекламы или самой HH.

n8n webhook на каждое событие без фильтрации.
В MVP‑1 «отправка события в n8n» для каждого сохранения или смены статуса может привести к шуму и излишнему сетевому трафику. Нужны конфигурируемые триггеры, а не поток.

Недооценена сложность «рекомендации резюме».
Без доступа к API HH резюме и без парсинга страницы резюме рекомендация будет либо статичной привязкой профиль→резюме, либо потребует отдельного AI‑анализа, что увеличит расход токенов.

Исправленная стратегия
Разделить MVP‑1 на MVP‑0 (Silent Observer) и MVP‑1 (Smart Saver).
MVP‑0: теневое извлечение данных на странице вакансии, локальное сохранение + ручные статусы. Никакого UI, кроме иконки расширения «работает/нет». Это позволяет отладить парсинг на реальных данных, собрать тестовые фикстуры, проверить отсутствие детекта.
MVP‑1: добавляем боковую панель с анализом, rule‑based score, AI‑кнопку, профили, генерацию письма, экспорт. Но без очереди откликов и без трогания страницы поиска.

MVP‑2 — поисковая выдача, но в два этапа: сначала подсветка на основе локальной БД (без изменения DOM HH), затем безопасные бейджи через изолированный слой.

MVP‑3/4 — отклики с полуавтоматикой: ручная подстановка письма по клику, вставка в форму с задержкой, только после явного действия пользователя на каждой странице. Никакой фоновой очереди.

MVP‑5 — работа с чатами и ответами HR: начинается только после накопления реального опыта использования, так как наиболее вариативен и чувствителен.

AI‑промпты минимальны, используют только очищенный текст, не отправляют HTML.

n8n webhook: в MVP‑1 только ручная отправка «отчёт по вакансии» по кнопке; автоматические события — начиная с MVP‑2, с настройками фильтрации.

Приватность: все разрешения объяснены пользователю при установке; дефолт — не отправлять ничего без действия.

MVP scope (пересмотренный)
MVP‑0: Silent Observer (неделя 1–2)
Извлечение данных со страницы вакансии HH: title, company, salary, city, remote/hybrid/office, experience, description, skills, source URL, vacancy ID.

Сохранение в локальную БД (IndexedDB).

Ручная установка статуса через popup (ограниченный набор: new, viewed, saved, rejected).

Экспорт CSV/JSON только сохранённых вакансий.

Значок расширения показывает количество сохранённых сегодня.

Никакого UI на странице, только иконка браузера.

MVP‑1: Smart Saver & Analyzer (неделя 3–5)
Боковая панель на странице вакансии (side panel через chrome.sidePanel или iframe в контент‑скрипте).

Rule‑based scoring (подробно ниже).

Кнопка «AI‑анализ» → отправка очищенного JSON в BYOK‑провайдер → отображение рекомендации, рисков, подходящего профиля.

Менеджер профилей (создание/редактирование) в dedicated странице расширения (chrome-extension://.../profiles.html).

Генерация сопроводительного письма по шаблону + AI‑адаптация (по кнопке).

Сохранение финального письма в карточку вакансии.

Расширенная статусная модель (все 11 статусов).

Ручной экспорт/импорт данных.

n8n webhook: только ручная отправка «поделиться вакансией» (payload: id, title, score, статус).

MVP‑2: Search Highlighter (неделя 6–8)
Контент‑скрипт на страницу поиска, читающий DOM и сверяющий с локальной БД.

Инжект минимальных CSS‑классов для подсветки строк (new/viewed/saved/applied/rejected/blacklist) через изолированный теневой DOM.

Score на карточке (иконка или маленький badge).

Быстрые действия: добавить в избранное/чёрный список через контекстное меню расширения или маленькую иконку.

Интеграция с n8n: автоматическая отправка события, когда пользователь вручную сохраняет вакансию с поиска (если включено).

MVP‑3: Semi‑Auto Apply (неделя 9–11)
На странице вакансии кнопка «Подготовить отклик».

AI‑генерация письма на основе выбранного профиля, шаблона и текста вакансии.

Показ предпросмотра письма в боковой панели с возможностью редактирования.

Кнопка «Вставить в форму» → скрипт находит поле ввода на HH и вставляет текст с имитацией печати (или через clipboard API с подтверждением пользователя).

Финальное подтверждение: пользователь сам нажимает «Отправить» в интерфейсе HH.

После отправки — смена статуса на «applied» локально и вебхук (по настройке).

Никакой автоматической отправки, никакой очереди.

MVP‑4: Apply Queue (неделя 12–14)
Страница расширения «Очередь откликов» — список вакансий со статусом «letter_ready» или «saved».

Для каждой — предпросмотр сгенерированного письма и кнопки «Открыть вакансию и вставить письмо», «Пропустить», «В чёрный список».

При клике «Открыть» — открывается новая вкладка с вакансией, и расширение в боковой панели показывает готовое письмо с кнопкой вставки.

Очередь не работает в фоне; каждая вакансия требует явного действия пользователя на её странице.

MVP‑5: HR Communication Hub (неделя 15+)
Парсинг страниц откликов и чатов HH (только по явному открытию пользователем).

Классификация ответов HR (положительный/отказ/тестовое/интервью) с помощью локального правила или AI по запросу.

Генерация ответа HR на основе шаблона.

Follow‑up reminders: локальные уведомления «напомнить через N дней».

Отправка событий в n8n/Telegram (только по триггерам, настраиваемым пользователем).

В этом этапе — только помощь в коммуникации, без автоматической отправки сообщений через расширение.

Архитектура
Стек: WXT (Manifest V3) + React (для side panel и страниц настроек) + Dexie (IndexedDB) + chrome.storage (настройки) + TypeScript.

Структура:

text
src/
  entrypoints/
    background.ts       // service worker, хаб сообщений, webhook-клиент, инициализация БД
    content-vacancy.ts  // контент-скрипт на страницу вакансии: парсинг, инжект боковой панели
    content-search.ts   // контент-скрипт на страницу поиска: чтение DOM, инжект подсветки
    sidepanel/          // React-приложение для боковой панели (vacancy analysis)
    dashboard/          // React-приложение для страниц расширения: profiles, queue, settings, export
    popup/              // маленький popup для быстрых действий (сохранить, статус)
  lib/
    parser-hh/          // извлечение данных со страницы вакансии, селекторы с fallback
    scorer/             // rule-based scoring engine
    ai/                 // клиент к OpenAI/OpenRouter/DeepSeek, шаблоны промптов
    db/                 // Dexie схемы, методы
    webhook/            // n8n клиент
    storage/            // абстракция над chrome.storage
    dom-utils/          // безопасная вставка UI, shadow DOM
  shared/
    types.ts            // общие типы
    constants.ts
Коммуникация:

content-script ↔ service worker: chrome.runtime.sendMessage

side panel / pages ↔ service worker: chrome.runtime.sendMessage

Данные хранятся локально в IndexedDB, доступ к ним из всех контекстов через service worker (или через Dexie с прямым доступом, синхронизация не требуется, т.к. один клиент).

Внедрение UI:

Боковая панель реализуется либо через chrome.sidePanel.open(), либо (как запасной вариант) через инжект фиксированного блока в body страницы с изоляцией через shadow DOM.

Подсветка на поиске: инжект минимального CSS и aria-атрибутов в строки результатов; никакого внедрения React‑компонентов в DOM HH.

Безопасность контент‑скриптов:

Парсинг только текста, никакого выполнения скриптов HH.

Все сетевые запросы к AI и webhook — только из service worker с валидацией payload.

Rule‑based Scoring Model (для HH‑вакансий)
Базовая оценка baseScore = 0..100. Критические флаги влияют на рекомендацию, но не обязательно на число.

Параметры из вакансии и профиля:

Совпадение по ключевым навыкам (вес 40)

Извлекаем hard skills из вакансии (например, Python, Docker, SQL).

Пересечение со skills выбранного профиля.

skillRatio = пересечение / навыки_вакансии.

Если < 0.3 → -10 к финальной рекомендации.

Соответствие опыта (вес 15)

Парсим строку опыта (например, "3–6 лет") в среднее число.

Сравниваем с профилем: если входит ±1 год → 15, если сильно меньше (кандидат junior, а требуют senior) → 0.

Если опыт не указан — нейтрально (10).

Зарплата (вес 10)

Если есть from/to, проверяем попадание в ожидания профиля (настраивается).

Отсутствие зарплаты — не штрафуем, но снижаем вес до 5.

Локация/удалёнка (вес 10)

Совпадение предпочтений профиля: remote/hybrid/office/city.

Несовпадение города без удалёнки → 0.

Тип занятости (вес 5)

Если извлечено (проектная/постоянная), сравниваем.

Качество описания (вес 10)

Штрафуем за: отсутствие обязанностей (только требования), очень короткий текст (<200 символов), наличие списков "требуется всё сразу".

Компания в чёрном списке (критический флаг) → сразу рекомендация «пропустить» или «черный список».

Дубликат/переопубликованная (если уже есть в БД с другим ID) — флаг «возможный дубликат».

Выходные метрики:

matchScore: число 0–100.

recommendation: apply / consider / skip.

riskFlags: массив строк (low_salary, vague_description, skill_gap, long_requirements, no_remote).

fitReasons: массив строк (skill_match_high, salary_good, remote_match).

AI Prompt Templates
Все промпты строятся на минимальном очищенном JSON, не содержат полного HTML и личной информации без согласия пользователя.

1. Анализ вакансии (режим byok, модель может быть легкой)
System prompt:

text
You are a career coach and job matching assistant. You will receive a JSON object with a job vacancy and a candidate profile. Analyze the match and return a JSON response with:
- fitScore (0-100)
- recommendation: "apply" | "consider" | "skip"
- fitReasons: string[]
- riskFlags: string[]
- suggestedProfile: string (name of best matching profile from user's list, or "none")
- coverLetterDraft: string (a draft tailored to the vacancy, max 800 characters, in the specified tone)
Respond only with valid JSON, no explanation.
User message:

json
{
  "vacancy": {
    "title": "...",
    "company": "...",
    "description": "... (cleaned text, truncated to 3000 chars)",
    "skills": [...],
    "experience": "...",
    "remote": "..."
  },
  "profile": {
    "name": "...",
    "skills": [...],
    "experienceYears": ...,
    "desiredRoles": [...]
  },
  "letterOptions": {
    "mode": "confident",
    "maxChars": 500
  }
}
Ответ ожидается:

json
{
  "fitScore": 82,

2. Генерация сопроводительного письма отдельно
Аналогичный системный промпт, но с акцентом на стиль и адаптацию под вакансию. В user message передаётся vacancy, profile, letterOptions. Ответ содержит только текст письма.

Снижение токенов: дефолтные промпты хранятся локально, пользователь может редактировать их в настройках.

UX/UI структура
Значок расширения (browser action):

Badge с количеством новых вакансий сегодня.

Popup: мини-дашборд (последние сохранённые вакансии, быстрые переходы в Saved/Queue/Profiles/Settings).

Страница вакансии:

Боковая панель (открывается кликом по кнопке на странице или автоматически при сохранении в настройках).

В панели: статус, rule‑based score (мгновенно), кнопка «AI анализ», рекомендации, генерация письма, предпросмотр, сохранение.

Инжектится только минимальная кнопка «открыть копилот» в тело страницы, если боковая панель не открыта.

Страница поиска:

Подсветка строк через CSS классы + небольшая иконка с score.

Контекстное меню расширения (right‑click) -> «Сохранить», «В чёрный список».

Страницы расширения (chrome-extension://...):

/profiles — CRUD профилей.

/saved — таблица всех вакансий с фильтрами, поиском, экспортом.

/queue — очередь откликов (с MVP‑4).

/settings — AI ключи, webhook URL, режимы писем, тогглы.

Onboarding: при первой установке — краткое объяснение разрешений, настройка профиля, BYOK (можно пропустить).

Security / Privacy модель
Минимальные разрешения:
"storage", "activeTab" (или "scripting" с host_permissions для HH.ru), "sidePanel".
Host permissions: https://hh.ru/* — только необходимый минимум, без all_urls.

Сетевые запросы:

К AI‑провайдеру — только через service worker, только по явному действию пользователя, payload содержит очищенный текст, без cookies/заголовков HH.

n8n webhook — отправка только тех полей, которые пользователь разрешил (по умолчанию: vacancyId, title, score, status). Никаких личных данных пользователя или полного текста вакансии.

Все запросы с Content-Type: application/json, поддержка HTTPS enforced.

Локальное хранение:

IndexedDB не шифруется (браузерное хранилище), поэтому в настройках строго не хранить сырые API ключи? API ключ хранится в chrome.storage.local (незашифрован, но пользователь предупреждён). Можно зашифровать через WebCrypto с мастер‑паролем (V2).

Данные вакансий хранятся локально, экспорт только вручную.

Защита от утечек:

При отправке AI промпта удаляются все поля, не указанные в шаблоне; проверка на наличие чувствительных паттернов (email, телефон) — игнорировать или предупреждать пользователя.

n8n webhook safety:

Только по HTTPS.

Signature verification (опционально, пользователь задаёт секрет в настройках, расширение добавляет заголовок X-Signature).

Rate limit локально: не более 1 запроса в 2 секунды.

# Что в концепции сильное

- **User‑controlled copilot**, а не бот. Никакой автоотправки без подтверждения.
- **Local‑first**, BYOK для AI, минимум данных наружу — отличный приватный дефолт.

# Что в концепции опасно или спорно

1. **Слишком много MVP‑1.**  
   Анализ, сохранение, AI‑генерация, экспорт, webhook, 10 статусов, rule‑based score, профили, шаблоны — это всё сразу в первой версии создаст высокую когнитивную нагрузку и длительный цикл разработки. Риск: разработчик утонет в фичах, а базовый парсинг HH ещё нестабилен.

2. **DOM‑мутации на странице поиска HH (MVP‑2) без фазы наблюдателя.**  
   HH.ru активно обновляет вёрстку, A/B‑тестирует, использует динамическую подгрузку. Без «тихого» периода сбора реальных селекторов и тестирования на разных аккаунтах можно получить неработающие бейджи или (хуже) срабатывание антибот‑скриптов.

3. **«Очередь откликов» в MVP‑3 и MVP‑4 может быть воспринята HH как автоматизированная активность.**  
   Даже с подтверждением, последовательная подстановка текста и клик «Отправить» с интервалами — паттерн, который системы защиты могут задетектировать. Надо проектировать с «человеческими» задержками и без инжекта в защищённые элементы.

4. **Нет чёткого разделения между «извлечением данных» и «модификацией страницы».**  
   Content script, который одновременно парсит вакансию и встраивает UI на страницу, может конфликтовать с CSP, блокировщиками рекламы или самой HH.

5. **n8n webhook на каждое событие без фильтрации.**  
   В MVP‑1 «отправка события в n8n» для каждого сохранения или смены статуса может привести к шуму и излишнему сетевому трафику. Нужны конфигурируемые триггеры, а не поток.

6. **Недооценена сложность «рекомендации резюме».**  
   Без доступа к API HH резюме и без парсинга страницы резюме рекомендация будет либо статичной привязкой профиль→резюме, либо потребует отдельного AI‑анализа, что увеличит расход токенов.

# Исправленная стратегия

- **Разделить MVP‑1 на MVP‑0 (Silent Observer) и MVP‑1 (Smart Saver).**  
  MVP‑0: теневое извлечение данных на странице вакансии, локальное сохранение + ручные статусы. Никакого UI, кроме иконки расширения «работает/нет». Это позволяет отладить парсинг на реальных данных, собрать тестовые фикстуры, проверить отсутствие детекта.  
  MVP‑1: добавляем боковую панель с анализом, rule‑based score, AI‑кнопку, профили, генерацию письма, экспорт. Но без очереди откликов и без трогания страницы поиска.

- **MVP‑2 — поисковая выдача, но в два этапа:** сначала подсветка на основе локальной БД (без изменения DOM HH), затем безопасные бейджи через изолированный слой.

- **MVP‑3/4 — отклики с полуавтоматикой:** ручная подстановка письма по клику, вставка в форму с задержкой, только после явного действия пользователя на каждой странице. Никакой фоновой очереди.

- **MVP‑5 — работа с чатами и ответами HR:** начинается только после накопления реального опыта использования, так как наиболее вариативен и чувствителен.

- **AI‑промпты минимальны, используют только очищенный текст, не отправляют HTML.**

- **n8n webhook:** в MVP‑1 только ручная отправка «отчёт по вакансии» по кнопке; автоматические события — начиная с MVP‑2, с настройками фильтрации.

- **Приватность:** все разрешения объяснены пользователю при установке; дефолт — не отправлять ничего без действия.

- Менеджер профилей (создание/редактирование) в dedicated странице расширения (chrome-extension://.../profiles.html).
- Генерация сопроводительного письма по шаблону + AI‑адаптация (по кнопке).
- Сохранение финального письма в карточку вакансии.
- Расширенная статусная модель (все 11 статусов).
- Ручной экспорт/импорт данных.
- n8n webhook: только ручная отправка «поделиться вакансией» (payload: id, title, score, статус).

### MVP‑2: Search Highlighter (неделя 6–8)
- Контент‑скрипт на страницу поиска, **читающий** DOM и сверяющий с локальной БД.
- Инжект минимальных CSS‑классов для подсветки строк (new/viewed/saved/applied/rejected/blacklist) через изолированный теневой DOM.
- Score на карточке (иконка или маленький badge).
- Быстрые действия: добавить в избранное/чёрный список через контекстное меню расширения или маленькую иконку.
- Интеграция с n8n: автоматическая отправка события, когда пользователь вручную сохраняет вакансию с поиска (если включено).

### MVP‑3: Semi‑Auto Apply (неделя 9–11)
- На странице вакансии кнопка «Подготовить отклик».
- AI‑генерация письма на основе выбранного профиля, шаблона и текста вакансии.
- Показ предпросмотра письма в боковой панели с возможностью редактирования.
- Кнопка «Вставить в форму» → скрипт находит поле ввода на HH и вставляет текст с имитацией печати (или через clipboard API с подтверждением пользователя).

- После отправки — смена статуса на «applied» локально и вебхук (по настройке).

### MVP‑5: HR Communication Hub (неделя 15+)

- Генерация ответа HR на основе шаблона.
- Follow‑up reminders: локальные уведомления «напомнить через N дней».
- Отправка событий в n8n/Telegram (только по триггерам, настраиваемым пользователем).
- **В этом этапе — только помощь в коммуникации, без автоматической отправки сообщений через расширение.**

Статусы (JobStatus): new, viewed, saved, rejected_by_me, letter_ready, applied, hr_replied, interview, test_task, rejected_by_company, offer, blacklist.

Rule‑based Scoring Model (для HH‑вакансий)
Базовая оценка baseScore = 0..100. Критические флаги влияют на рекомендацию, но не обязательно на число.

Параметры из вакансии и профиля:

Совпадение по ключевым навыкам (вес 40)

Извлекаем hard skills из вакансии (например, Python, Docker, SQL).

Пересечение со skills выбранного профиля.

skillRatio = пересечение / навыки_вакансии.

Если < 0.3 → -10 к финальной рекомендации.

Соответствие опыта (вес 15)

Парсим строку опыта (например, "3–6 лет") в среднее число.

Сравниваем с профилем: если входит ±1 год → 15, если сильно меньше (кандидат junior, а требуют senior) → 0.

Если опыт не указан — нейтрально (10).

Зарплата (вес 10)

Если есть from/to, проверяем попадание в ожидания профиля (настраивается).

Отсутствие зарплаты — не штрафуем, но снижаем вес до 5.

Локация/удалёнка (вес 10)

Совпадение предпочтений профиля: remote/hybrid/office/city.

Несовпадение города без удалёнки → 0.

Тип занятости (вес 5)

Если извлечено (проектная/постоянная), сравниваем.

Качество описания (вес 10)

Штрафуем за: отсутствие обязанностей (только требования), очень короткий текст (<200 символов), наличие списков "требуется всё сразу".

Компания в чёрном списке (критический флаг) → сразу рекомендация «пропустить» или «черный список».

Дубликат/переопубликованная (если уже есть в БД с другим ID) — флаг «возможный дубликат».

Выходные метрики:

matchScore: число 0–100.

recommendation: apply / consider / skip.

riskFlags: массив строк (low_salary, vague_description, skill_gap, long_requirements, no_remote).

fitReasons: массив строк (skill_match_high, salary_good, remote_match).

Только по HTTPS.

Signature verification (опционально, пользователь задаёт секрет в настройках, расширение добавляет заголовок X-Signature).

Rate limit локально: не более 1 запроса в 2 секунды.

# Executive Summary

Концепция **CareerSignal HH Copilot** — одна из самых продуманных начальных спецификаций «не‑спам»‑расширения для рынка труда, которые я видел.  
Она правильно ставит во главу угла безопасность пользователя, приватность, контроль над автоматизацией и архитектурную чистоту.  
Тем не менее есть несколько противоречий между «быстрым MVP» и «глубокой автоматизацией», а также недооценённые риски, связанные с DOM‑мутациями, антибот‑защитой и объёмом первой версии.

Ниже — критический аудит, исправленная продуктовая стратегия, архитектура, модели данных, AI‑промпты, планы безопасности и роадмап, оптимизированный под одного разработчика.

---

# Что в концепции сильное

- **User‑controlled copilot**, а не бот. Никакой автоотправки без подтверждения.
- **Local‑first**, BYOK для AI, минимум данных наружу — отличный приватный дефолт.
- **Manifest V3 + WXT + TypeScript + Dexie** — современный, расширяемый стек.
- **Гибридный скоринг**: правило + AI по требованию — экономит деньги и повышает скорость.
- **Профили кандидата и шаблоны писем** — сразу закладывается персонализация, а не универсальный спам.
- **MVP строго на HH.ru**, но архитектура рассчитана на будущие сайты — правильный баланс.
- **Статусная модель** почти полностью покрывает реальный флоу кандидата.
- Отказ от обязательного бэкенда — снижает порог входа и риск утечек.
- Правильные приоритеты: не получить бан → качество откликов → скорость MVP.

---

# Что в концепции опасно или спорно

1. **Слишком много MVP‑1.**  
   Анализ, сохранение, AI‑генерация, экспорт, webhook, 10 статусов, rule‑based score, профили, шаблоны — это всё сразу в первой версии создаст высокую когнитивную нагрузку и длительный цикл разработки. Риск: разработчик утонет в фичах, а базовый парсинг HH ещё нестабилен.

2. **DOM‑мутации на странице поиска HH (MVP‑2) без фазы наблюдателя.**  
   HH.ru активно обновляет вёрстку, A/B‑тестирует, использует динамическую подгрузку. Без «тихого» периода сбора реальных селекторов и тестирования на разных аккаунтах можно получить неработающие бейджи или (хуже) срабатывание антибот‑скриптов.

3. **«Очередь откликов» в MVP‑3 и MVP‑4 может быть воспринята HH как автоматизированная активность.**  
   Даже с подтверждением, последовательная подстановка текста и клик «Отправить» с интервалами — паттерн, который системы защиты могут задетектировать. Надо проектировать с «человеческими» задержками и без инжекта в защищённые элементы.

4. **Нет чёткого разделения между «извлечением данных» и «модификацией страницы».**  
   Content script, который одновременно парсит вакансию и встраивает UI на страницу, может конфликтовать с CSP, блокировщиками рекламы или самой HH.

5. **n8n webhook на каждое событие без фильтрации.**  
   В MVP‑1 «отправка события в n8n» для каждого сохранения или смены статуса может привести к шуму и излишнему сетевому трафику. Нужны конфигурируемые триггеры, а не поток.

6. **Недооценена сложность «рекомендации резюме».**  
   Без доступа к API HH резюме и без парсинга страницы резюме рекомендация будет либо статичной привязкой профиль→резюме, либо потребует отдельного AI‑анализа, что увеличит расход токенов.

---

# Исправленная стратегия

- **Разделить MVP‑1 на MVP‑0 (Silent Observer) и MVP‑1 (Smart Saver).**  
  MVP‑0: теневое извлечение данных на странице вакансии, локальное сохранение + ручные статусы. Никакого UI, кроме иконки расширения «работает/нет». Это позволяет отладить парсинг на реальных данных, собрать тестовые фикстуры, проверить отсутствие детекта.  
  MVP‑1: добавляем боковую панель с анализом, rule‑based score, AI‑кнопку, профили, генерацию письма, экспорт. Но без очереди откликов и без трогания страницы поиска.

- **MVP‑2 — поисковая выдача, но в два этапа:** сначала подсветка на основе локальной БД (без изменения DOM HH), затем безопасные бейджи через изолированный слой.

- **MVP‑3/4 — отклики с полуавтоматикой:** ручная подстановка письма по клику, вставка в форму с задержкой, только после явного действия пользователя на каждой странице. Никакой фоновой очереди.

- **MVP‑5 — работа с чатами и ответами HR:** начинается только после накопления реального опыта использования, так как наиболее вариативен и чувствителен.

- **AI‑промпты минимальны, используют только очищенный текст, не отправляют HTML.**

- **n8n webhook:** в MVP‑1 только ручная отправка «отчёт по вакансии» по кнопке; автоматические события — начиная с MVP‑2, с настройками фильтрации.

- **Приватность:** все разрешения объяснены пользователю при установке; дефолт — не отправлять ничего без действия.

### MVP‑0: Silent Observer (неделя 1–2)

- Боковая панель на странице вакансии (side panel через chrome.sidePanel или iframe в контент‑скрипте).
- Rule‑based scoring (подробно ниже).
- Кнопка «AI‑анализ» → отправка очищенного JSON в BYOK‑провайдер → отображение рекомендации, рисков, подходящего профиля.
- Менеджер профилей (создание/редактирование) в dedicated странице расширения (chrome-extension://.../profiles.html).
- Генерация сопроводительного письма по шаблону + AI‑адаптация (по кнопке).
- Сохранение финального письма в карточку вакансии.
- Расширенная статусная модель (все 11 статусов).
- Ручной экспорт/импорт данных.
- n8n webhook: только ручная отправка «поделиться вакансией» (payload: id, title, score, статус).

### MVP‑2: Search Highlighter (неделя 6–8)
- Контент‑скрипт на страницу поиска, **читающий** DOM и сверяющий с локальной БД.
- Инжект минимальных CSS‑классов для подсветки строк (new/viewed/saved/applied/rejected/blacklist) через изолированный теневой DOM.
- Score на карточке (иконка или маленький badge).
- Быстрые действия: добавить в избранное/чёрный список через контекстное меню расширения или маленькую иконку.
- Интеграция с n8n: автоматическая отправка события, когда пользователь вручную сохраняет вакансию с поиска (если включено).

### MVP‑3: Semi‑Auto Apply (неделя 9–11)
- На странице вакансии кнопка «Подготовить отклик».
- AI‑генерация письма на основе выбранного профиля, шаблона и текста вакансии.
- Показ предпросмотра письма в боковой панели с возможностью редактирования.
- Кнопка «Вставить в форму» → скрипт находит поле ввода на HH и вставляет текст с имитацией печати (или через clipboard API с подтверждением пользователя).
- Финальное подтверждение: пользователь сам нажимает «Отправить» в интерфейсе HH.
- После отправки — смена статуса на «applied» локально и вебхук (по настройке).
- **Никакой автоматической отправки, никакой очереди.**

### MVP‑4: Apply Queue (неделя 12–14)
- Страница расширения «Очередь откликов» — список вакансий со статусом «letter_ready» или «saved».
- Для каждой — предпросмотр сгенерированного письма и кнопки «Открыть вакансию и вставить письмо», «Пропустить», «В чёрный список».
- При клике «Открыть» — открывается новая вкладка с вакансией, и расширение в боковой панели показывает готовое письмо с кнопкой вставки.
- Очередь не работает в фоне; каждая вакансия требует явного действия пользователя на её странице.

### MVP‑5: HR Communication Hub (неделя 15+)
- Парсинг страниц откликов и чатов HH (только по явному открытию пользователем).
- Классификация ответов HR (положительный/отказ/тестовое/интервью) с помощью локального правила или AI по запросу.
- Генерация ответа HR на основе шаблона.
- Follow‑up reminders: локальные уведомления «напомнить через N дней».
- Отправка событий в n8n/Telegram (только по триггерам, настраиваемым пользователем).
- **В этом этапе — только помощь в коммуникации, без автоматической отправки сообщений через расширение.**

**Коммуникация:**  
- content-script ↔ service worker: `chrome.runtime.sendMessage`  
- side panel / pages ↔ service worker: `chrome.runtime.sendMessage`  
- Данные хранятся локально в IndexedDB, доступ к ним из всех контекстов через service worker (или через Dexie с прямым доступом, синхронизация не требуется, т.к. один клиент).

**Внедрение UI:**  
- Боковая панель реализуется либо через `chrome.sidePanel.open()`, либо (как запасной вариант) через инжект фиксированного блока в body страницы с изоляцией через shadow DOM.  
- Подсветка на поиске: инжект минимального CSS и aria-атрибутов в строки результатов; никакого внедрения React‑компонентов в DOM HH.

**Безопасность контент‑скриптов:**  
- Парсинг только текста, никакого выполнения скриптов HH.  
- Все сетевые запросы к AI и webhook — только из service worker с валидацией payload.

Rule‑based Scoring Model (для HH‑вакансий)
Базовая оценка baseScore = 0..100. Критические флаги влияют на рекомендацию, но не обязательно на число.

Параметры из вакансии и профиля:

Совпадение по ключевым навыкам (вес 40)

Извлекаем hard skills из вакансии (например, Python, Docker, SQL).

Пересечение со skills выбранного профиля.

skillRatio = пересечение / навыки_вакансии.

Если < 0.3 → -10 к финальной рекомендации.

Соответствие опыта (вес 15)

Парсим строку опыта (например, "3–6 лет") в среднее число.

Сравниваем с профилем: если входит ±1 год → 15, если сильно меньше (кандидат junior, а требуют senior) → 0.

Если опыт не указан — нейтрально (10).

Зарплата (вес 10)

Если есть from/to, проверяем попадание в ожидания профиля (настраивается).

Отсутствие зарплаты — не штрафуем, но снижаем вес до 5.

Локация/удалёнка (вес 10)

Совпадение предпочтений профиля: remote/hybrid/office/city.

Несовпадение города без удалёнки → 0.

Тип занятости (вес 5)

Если извлечено (проектная/постоянная), сравниваем.

Качество описания (вес 10)

Штрафуем за: отсутствие обязанностей (только требования), очень короткий текст (<200 символов), наличие списков "требуется всё сразу".

Компания в чёрном списке (критический флаг) → сразу рекомендация «пропустить» или «черный список».

Дубликат/переопубликованная (если уже есть в БД с другим ID) — флаг «возможный дубликат».

Выходные метрики:

matchScore: число 0–100.

recommendation: apply / consider / skip.

riskFlags: массив строк (low_salary, vague_description, skill_gap, long_requirements, no_remote).

fitReasons: массив строк (skill_match_high, salary_good, remote_match).

Security / Privacy модель
Минимальные разрешения:
"storage", "activeTab" (или "scripting" с host_permissions для HH.ru), "sidePanel".
Host permissions: https://hh.ru/* — только необходимый минимум, без all_urls.

Сетевые запросы:

К AI‑провайдеру — только через service worker, только по явному действию пользователя, payload содержит очищенный текст, без cookies/заголовков HH.

n8n webhook — отправка только тех полей, которые пользователь разрешил (по умолчанию: vacancyId, title, score, status). Никаких личных данных пользователя или полного текста вакансии.

Все запросы с Content-Type: application/json, поддержка HTTPS enforced.

Локальное хранение:

IndexedDB не шифруется (браузерное хранилище), поэтому в настройках строго не хранить сырые API ключи? API ключ хранится в chrome.storage.local (незашифрован, но пользователь предупреждён). Можно зашифровать через WebCrypto с мастер‑паролем (V2).

Данные вакансий хранятся локально, экспорт только вручную.

Защита от утечек:

При отправке AI промпта удаляются все поля, не указанные в шаблоне; проверка на наличие чувствительных паттернов (email, телефон) — игнорировать или предупреждать пользователя.

n8n webhook safety:

Только по HTTPS.

Signature verification (опционально, пользователь задаёт секрет в настройках, расширение добавляет заголовок X-Signature).

Rate limit локально: не более 1 запроса в 2 секунды.

Никогда не использовать неофициальные API HH, только DOM.

Не имитировать скроллинг или клики по элементам HH от лица расширения.

Каждое изменение на странице инициируется пользователем (клик, горячая клавиша).

Пользователь остаётся полностью ответственным за финальное действие «Отправить отклик».

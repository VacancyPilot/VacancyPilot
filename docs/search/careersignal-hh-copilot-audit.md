# CareerSignal HH Copilot — критический аудит концепции и архитектура MVP

---

## Executive Summary

Концепция в целом здравая: local-first, BYOK, гибридный scoring, поэтапная автоматизация с ручным подтверждением — это правильная философия для соло-разработчика, который не хочет получить бан на единственной реально важной площадке (HH.ru) и не хочет городить backend ради MVP.

Но контекст рынка сместился за последние месяцы сильнее, чем учитывает ТЗ. **15 декабря 2025 hh.ru закрыл публичное API для соискателей** именно для борьбы с автооткликами и сервисами автоматизации (Habr, декабрь 2025). Это значит:

1. Любая интеграция строго через DOM той страницы, которую реально открыл пользователь в своей аутентифицированной сессии — это теперь не «один из вариантов», а единственный легитимный путь. Прямые обращения к api.hh.ru/любым закрытым эндпоинтам исключены полностью, даже «только на чтение».
2. Сама причина закрытия API — массовые автоотклики. Это означает, что MVP-3 (полуавтоотклик) и MVP-4 (очередь откликов) находятся ровно в той зоне, которую hh.ru сейчас активно прижимает. Их нельзя считать «следующим логичным шагом» — это отдельный, повышенно рискованный модуль, который нужно явно отделить от core-продукта.
3. Защита hh.ru — это не только rate-limit. Подтверждено использование капчи и блокировок по поведенческим/identity-сигналам, а не только по IP. «Полуавто с подтверждением» снижает юридический и репутационный риск, но не снимает технический риск детекта автоматизации на уровне DOM-событий (synthetic vs trusted events) и паттернов поведения.

**Главный вывод**: сузить MVP-1 до того, что само по себе уже продаваемый продукт без всякого автоматического взаимодействия с формами — анализ, скоринг, письма, трекер, экспорт, n8n-события. Это закрывает 80% ценности при 5% риска. MVP-3/4 — отдельный, явно помеченный как experimental, выключаемый по умолчанию модуль, к которому стоит возвращаться только после того, как стало понятно, как именно hh.ru сейчас детектит автоматизацию (после декабрьского ужесточения это движущаяся цель).

Приоритет «максимальная автоматизация» из исходного ТЗ стоит убрать целиком — он логически противоречит приоритету №1 «не получить бан». Цель — не объём действий, а неотличимость поведения пользователя+расширения от поведения обычного внимательного кандидата.

## 1. Что в концепции сильное

- **Local-first + экспорт CSV/JSON + BYOK** — нет vendor lock-in, нет необходимости поднимать и платить за backend в MVP, данные физически остаются у пользователя. Хорошо как для приватности, так и для скорости разработки соло.
- **Гибридный scoring** (мгновенный rule-based + AI по кнопке) — экономит токены/деньги, работает и без интернета/ключа, даёт расширению ценность «из коробки» даже для пользователя без AI-подписки.
- **Поэтапная автоматизация** (ручной → полуавто → очередь с подтверждением, без полной автоотправки) — правильное направление мысли, проблема только в том, что после декабря 2025 даже «полуавто» требует пересмотра рисков (см. раздел Risks).
- **Profile Manager с несколькими специализациями** — реалистично отражает то, что один кандидат сейчас откликается под разными позиционированиями (AI Engineer / Automation / CRM / Systems Analyst), что прямо соответствует реальной ситуации мультидисциплинарного консультанта.
- **Письма как отдельная сильная фича**: шаблоны + AI-адаптация + библиотека удачных писем + жёсткие constraints (без emoji/markdown/лимит символов) — это то, что реально отличает инструмент от спам-бота и закрывает боль, которую автооткликеры не решают вообще. Это самая безопасная и при этом самая ценная часть продукта — стоит сделать её "hero feature" MVP-1, а не довесок.
- **Узкий скоуп страниц в MVP-1** (вакансия + поиск) — правильно, не пытается охватить отклики/чаты/приглашения сразу.
- **n8n как шина событий** вместо встроенного бэкенда — снимает с разработчика необходимость держать сервер, перекладывает уведомления/интеграции на инструмент, который пользователь и так умеет готовить.

## 2. Что в концепции опасно или спорно

1. **Закрытие API hh.ru (дек. 2025) меняет риск-профиль MVP-3/4.** Изначальное ТЗ трактует «полуавто с подтверждением» как безопасный средний вариант между крайностями. После декабрьского ужесточения это уже не нейтральная зона — это именно тот паттерн поведения, который hh.ru целенаправленно ловит. Нужно явно понизить статус этих MVP до «experimental, off by default».

2. **Противоречие приоритетов**: «не получить бан» — приоритет №1, «максимальная автоматизация» — приоритет №6 в том же списке. Если автоматизация вообще в списке целей продукта, разработчик рано или поздно столкнётся с выбором между ними при дизайне queue/auto-fill. Лучше явно вычеркнуть «максимальная автоматизация» из целей и заменить на «неотличимость от поведения живого пользователя».

3. **Synthetic DOM events при автоподстановке в форму (MVP-3).** Если расширение программно устанавливает `value` поля и диспатчит `input`/`change` события через JS, такие события в браузере помечены `isTrusted: false`. Если HH (или его антибот-слой) проверяет этот флаг или паттерн событий (отсутствие keydown/focus/blur последовательности, нулевая задержка между полями) — авто-заполнение детектируется даже при том, что финальный клик "Откликнуться" делает сам пользователь. Это нужно явно протестировать на тестовом аккаунте, а не считать решённым по дизайну.

4. **BYOK-ключи хранятся в `chrome.storage`, который не является secure storage.** Любой, у кого есть доступ к профилю Chrome (или вредоносное расширение с подходящими permissions), потенциально может прочитать ключ. Это не баг архитектуры — это системное ограничение всех browser extensions, но в ТЗ это нигде не проговорено как риск, который нужно явно показать пользователю при онбординге.

5. **Debug-режим с полным HTML** рискует случайно утащить в локальное хранилище PII третьих лиц (контакты HR, иногда телефон/email прямо в тексте вакансии) или служебные токены/идентификаторы, присутствующие в DOM. Даже «только локально» — это смешивание модели данных. Нужен redaction-фильтр даже в debug-режиме, а не просто флаг «вкл/выкл».

6. **Порог "сильная вакансия" (Match ≥ 85)** заявлен как абсолютное число с первого дня, но rule-based скоринг без калибровки на реальных данных конкретного пользователя почти гарантированно даст или слишком много, или слишком мало срабатываний в первую неделю. Нужен period калибровки и возможность пользователю подвинуть порог, иначе первая же неделя — это либо спам в Telegram, либо тишина, которая выглядит как «не работает».

7. **Нет описания, как разрешается конфликт источников истины.** Пользователь может откликнуться вручную прямо на hh.ru, минуя очередь расширения. Локальный статус разойдётся с реальностью. Нужен пассивный re-sync: расширение должно уметь читать бейдж «Вы откликались» / «Резюме просмотрено» с самой страницы HH и обновлять локальный статус, а не доверять только своим записям.

8. **"HR ответил" как n8n-событие** относится по факту к MVP-5 (чаты/приглашения), но в перечне MVP-1 функций оно не исключено явно — стоит прямо зафиксировать, что в MVP-1 доступны только `application_sent`, `strong_job_found`, `daily_summary`.

9. **Множественные резюме на HH** не учтены в Data model — у пользователя на HH обычно несколько резюме под разные позиционирования. "Рекомендация резюме" должна мапиться на конкретный `hhResumeId`, иначе рекомендация остаётся абстрактной и бесполезной в момент реального отклика.

10. **Нет non-goals / explicit "что не делаем"** — без этого списка scope creep гарантирован уже на MVP-2 (badges легко тянут за собой "а давайте ещё и фильтры", "а давайте ещё агрегацию по нескольким вкладкам").

## 3. Исправленная стратегия

**Позиционирование**: не "инструмент для большего числа откликов", а "copilot для меньшего числа, но более точных откликов, без риска бана". Это автоматически снижает требования к степени автоматизации и делает рискованные модули (MVP-3/4) опциональными, а не central value prop.

**Изменение порядка приоритетов** (вместо исходного списка из 6 пунктов):

1. Не получить бан / не нарушить ToS.
2. Качество отклика (письма, скоринг, релевантность).
3. Приватность и контроль данных пользователя.
4. Быстрый MVP с минимальными расходами на API.
5. Архитектурная чистота (адаптеры под будущие сайты).
6. ~~Максимальная автоматизация~~ → **Аутентичность поведения** (расширение усиливает решения пользователя, не имитирует его действия).

**Структурное решение**: разделить продукт на два слоя.

- **Core (MVP-1/2)** — read-only relative к HH: парсинг, скоринг, письма, трекер, экспорт, события. Никакого взаимодействия с формами HH. Это можно публиковать в Chrome Web Store без особых вопросов от ревью, и это безопасно для аккаунта пользователя по определению — расширение ничего не отправляет на HH от его имени.
- **Labs (MVP-3/4, опционально)** — semi-auto reply assist. Включается отдельным тумблером с явным предупреждением о текущей политике hh.ru, дневным лимитом действий, случайной задержкой между действиями, обязательным финальным кликом пользователя по нативной кнопке HH (не по кнопке расширения). Этот слой стоит делать после того, как core уже используется и даёт обратную связь, а не одновременно.

## 4. MVP Scope

### MVP-1 — Core (дополненный)

- Парсинг страницы вакансии HH: title, company, salary (+currency, gross/net), city, формат работы, опыт, описание (clean text), skills, source URL, hh vacancy id.
- Локальное сохранение (Dexie/IndexedDB), полный жизненный цикл статусов.
- Rule-based score (мгновенно, без сети) + breakdown по компонентам.
- AI-анализ по кнопке (BYOK): fit reasons, risk flags, рекомендация профиля, recommendation apply/consider/skip.
- Генерация письма (шаблон + AI-адаптация), сохранение финальной версии.
- **Пассивная синхронизация статуса** с тем, что реально показывает HH на странице (badge "Вы откликались" и т.п.) — добавлено к исходному скоупу, закрывает риск №7 выше.
- **Экран онбординга / consent**, явно показывающий, какие данные куда уходят (HH → локально; локально → AI по явному клику; локально → n8n по событию) — добавлено, нужно и для доверия, и для прохождения ревью стора.
- Экспорт CSV/JSON.
- n8n webhook: `application_status_saved`, `strong_job_found`, `daily_summary` (без `hr_replied` — это MVP-5).

### MVP-2 — Search-страница

- Бейджи на карточках вакансий (статус + score).
- Подсветка по статусу (new/viewed/saved/applied/rejected/blacklist).
- Быстрые действия из списка (save/skip/blacklist).
- Добавление в очередь (без обработки очереди — это MVP-4).

### MVP-3 — Semi-auto reply (Labs, off by default)

- Подготовка письма под конкретную вакансию.
- Рекомендация конкретного резюме (`hhResumeId`).
- Подстановка текста: предпочтительно через **копирование в буфер обмена** + подсветка нужного поля, а не программная запись в DOM-поле. Если делается прямая запись в поле — обязательно протестировать на тестовом/некритичном аккаунте, не на основном.
- Финальное подтверждение — обязательно нативный клик пользователя по родной кнопке HH, расширение не трогает кнопку отправки.

### MVP-4 — Очередь (Labs, off by default)

- Очередь с ручной обработкой по одной вакансии за раз (никаких "отправить все 20").
- Дневной лимит действий, настраиваемый пользователем, с разумным дефолтом (низким).
- Случайный интервал между обработкой элементов очереди (не строго последовательно/мгновенно).
- После действия — обновление локального статуса + webhook.

### MVP-5 — Отклики/приглашения/чаты HR

- Классификация ответов HR (rule-based + AI).
- Генерация ответа HR (черновик, не автоотправка).
- Follow-up reminders.
- Telegram/n8n уведомления о `hr_replied`.

### Чего точно не делать в MVP-1 (и в целом до отдельного решения)

- Никаких прямых обращений к api.hh.ru или любым неофициальным эндпоинтам — после закрытия публичного API в декабре 2025 это прямой путь к бану.
- Никакой полностью автоматической отправки откликов без клика пользователя.
- Никакой эмуляции капчи/обхода защит, никакой ротации аккаунтов/IP/прокси.
- Никакого фонового polling HH, пока расширение/браузер открыт, но пользователь не на странице HH (это уже похоже на бот-активность по паттерну запросов).
- Никакой отправки полного резюме/PII в AI без точечного, видимого пользователю согласия на каждое поле (не общий чекбокс один раз при установке).
- Никакой телеметрии в адрес разработчика расширения по умолчанию (если решите добавить — opt-in и явно описать, иначе это подрывает заявленную приватность продукта).
- Никакого "хранить весь HTML по умолчанию" — только в debug-режиме, и то с redaction.

## 5. Архитектура

**Стек**: WXT + Manifest V3 + TypeScript + Solid.js + Dexie (IndexedDB) + chrome.storage (только настройки, не секреты).

```
┌─────────────────────────────────────────────────────────────┐
│  hh.ru (вакансия / поиск)                                    │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ Content Script (per-route extractor, адаптер HHAdapter)│   │
│  │  - читает уже отрендеренный DOM, никаких своих fetch   │    │
│  │  - извлекает RawVacancyDTO / RawSearchListDTO          │    │
│  │  - рисует бейджи/side panel inline                     │    │
│  └───────────────────┬──────────────────────────────────┘    │
└──────────────────────┼────────────────────────────────────────┘
                        │ chrome.runtime messaging (typed RPC)
┌──────────────────────▼────────────────────────────────────────┐
│ Background Service Worker (MV3, event-driven, не персистентный)│
│  - нормализация DTO → Job/Company                              │
│  - Dexie: запись/чтение/миграции схемы                         │
│  - rule-based scoring engine                                   │
│  - AI client (fetch к OpenAI/DeepSeek/OpenRouter, BYOK)         │
│  - n8n webhook dispatcher                                       │
│  - chrome.alarms → ежедневная сводка                            │
│  - badge/icon state                                              │
└──────────────────────┬────────────────────────────────────────┘
                        │
        ┌───────────────┼───────────────────┐
        ▼               ▼                   ▼
   Popup (Solid)   Side Panel (Solid)   Options/Dashboard (Solid, tab)
   - сводка дня    - companion view     - таблица заявок (kanban/list)
   - быстрый CTA   - score breakdown    - Profile Manager
                   - actions на текущей - Settings (AI key, n8n url)
                     вакансии           - Export CSV/JSON
                                        - Privacy controls (что шарим)
```

**Адаптер-слой под будущие сайты** (минимальная инвестиция сейчас, чтобы не переписывать в V2):

```typescript
interface SiteAdapter {
  siteId: SourceSite;
  matchUrl(url: string): 'vacancy' | 'search' | null;
  extractVacancy(doc: Document): RawVacancyDTO | null;
  extractSearchList(doc: Document): RawSearchItemDTO[];
  // MVP-3/4 only, опционально не реализован для новых сайтов
  fillApplicationForm?(doc: Document, payload: FillPayload): boolean;
}
```
В MVP-1 реализован только `HHAdapter`. LinkedIn/Indeed/Djinni и т.д. — просто новые реализации интерфейса в V2, без переписывания background/UI слоя.

**Permissions (MV3)**: `storage`, `scripting` (только если нужна динамическая инъекция), `alarms`; host permissions ограничены `*://*.hh.ru/*` (конкретные поддомены, не `<all_urls>`). Намеренно **не используем** `webRequest`/`webRequestBlocking` — нам не нужно перехватывать сетевые запросы, только читать уже отрисованный DOM.

## 6. Data Model

```typescript
type SourceSite = 'hh.ru' | 'linkedin' | 'indeed' | 'glassdoor' | 'greenhouse'
                 | 'lever' | 'workday' | 'djinni' | 'rabota.md';

type JobStatus =
  | 'new' | 'viewed' | 'saved' | 'rejected_by_me' | 'letter_ready'
  | 'applied' | 'hr_replied' | 'interview' | 'test_task'
  | 'rejected_by_company' | 'offer' | 'blacklist';

interface Company {
  id: string;
  name: string;
  hhCompanyId?: string;
  industry?: string;
  blacklisted: boolean;
  blacklistReason?: string;
  notes?: string;
  createdAt: string;
  updatedAt: string;
}

interface ScoreBreakdown {
  skillMatch: number;       // 0-100, вес 35%
  seniorityMatch: number;   // 0-100, вес 15%
  salaryFit: number;        // 0-100, вес 15%
  locationFit: number;      // 0-100, вес 10%
  freshness: number;        // 0-100, вес 5%
  riskPenalty: number;      // 0..-20
  total: number;            // итоговый 0-100
}

interface AIAnalysis {
  modelUsed: string;
  fitReasons: string[];
  riskFlags: string[];
  recommendation: 'apply' | 'consider' | 'skip';
  recommendedProfileId?: string;
  generatedAt: string;
}

interface Job {
  id: string;
  sourceSite: SourceSite;
  sourceVacancyId: string;
  sourceUrl: string;
  title: string;
  companyId: string;
  salaryMin?: number;
  salaryMax?: number;
  salaryCurrency?: string;
  salaryGross?: boolean;
  city?: string;
  workMode: 'remote' | 'hybrid' | 'office' | 'unknown';
  experienceLevel?: string;
  descriptionClean: string;
  descriptionRawHtml?: string;   // только debug-режим, redacted
  skills: string[];
  postedAt?: string;
  fetchedAt: string;
  status: JobStatus;
  ruleScore: number;
  ruleScoreBreakdown: ScoreBreakdown;
  aiAnalysis?: AIAnalysis;
  recommendedProfileId?: string;
  createdAt: string;
  updatedAt: string;
}

interface Profile {
  id: string;
  name: string;                 // "AI Engineer / LLM / RAG"
  keywords: string[];
  mustHaveSkills: string[];
  niceToHaveSkills: string[];
  salaryExpectationMin?: number;
  salaryExpectationCurrency?: string;
  preferredWorkMode?: ('remote' | 'hybrid' | 'office')[];
  linkedResumeId?: string;
  createdAt: string;
  updatedAt: string;
}

interface Resume {
  id: string;
  profileId: string;
  hhResumeId?: string;          // маппинг на реальное резюме на HH
  title: string;
  highlightsText: string;       // для AI-промптов писем
  createdAt: string;
  updatedAt: string;
}

interface CoverLetter {
  id: string;
  jobId: string;
  profileId: string;
  mode: 'tg_short' | 'hh_standard' | 'confident' | 'very_short' | 'en';

  isFinal: boolean;
  source: 'template' | 'ai' | 'manual_edit';

  updatedAt: string;
}
```

## 7. Scoring Model (rule-based)

| Компонент | Вес | Логика |
|---|---|---|
| Skill match | 35% | пересечение `mustHaveSkills`/`niceToHaveSkills` профиля с навыками/описанием вакансии. Если 0 совпадений по must-have — компонент = 0 независимо от остального (hard override). |
| Seniority match | 15% | сравнение `experienceLevel` вакансии с профилем кандидата. |
| Salary fit | 15% | пересечение вилки вакансии с `salaryExpectationMin`. Если вилка не указана — нейтральные 50 + risk flag "зарплата не указана". |
| Location/формат | 10% | remote/hybrid/office вакансии vs `preferredWorkMode` профиля. |
| Freshness | 5% | штраф, если вакансия не обновлялась >30 дней. |
| Risk penalty | 0..-20 | вычитается за паттерны риска (ниже). |

**Типовые risk flags для CIS-рынка** (детектируются по тексту описания):

- "сдельная/сдельно-премиальная оплата" без указания базовой ставки;
- "оплата по результатам собеседования" / зарплата скрыта без объяснения;
- явное несоответствие требований и заявленного грейда (например "опыт от 5 лет" в вакансии "Junior");
- "разъездной характер работы" при заявленном remote;
- описание состоит преимущественно из soft-skills общих фраз без конкретики по стеку (низкая информативность → ниже уверенность скоринга, не обязательно риск, но стоит флагать отдельно как "low_signal");
- одна и та же компания публикует много похожих generic-вакансий одновременно (похоже на кадровое агентство/конвейер, не обязательно плохо, но влияет на ожидания по follow-up).

`total = clamp(0, 100, skillMatch*0.35 + seniorityMatch*0.15 + salaryFit*0.15 + locationFit*0.10 + freshness*0.05 + riskPenalty)`

Порог "сильная вакансия" (по умолчанию ≥85) должен быть настраиваемым в Settings с первого дня — без локальной калибровки на реальном потоке вакансий пользователя число 85 почти наверняка не будет соответствовать ожидаемой частоте срабатываний.

  - Export (CSV/JSON).

## 10. Security / Privacy

| Permission | Зачем | Риск, если расширить |
|---|---|---|
| `storage` | Dexie настройки, ключи, кэш | — |
| `scripting` | динамическая инъекция content script | избегать `<all_urls>`, держать host_permissions узкими |
| `alarms` | ежедневная сводка | — |
| host: `*://*.hh.ru/*` | чтение DOM страниц вакансии/поиска | не расширять на другие домены в MVP-1 |

**Ключевые принципы**:

- AI-ключ хранится в `chrome.storage.local` — это **не secure vault**, пользователь должен быть явно предупреждён при онбординге (любой с доступом к профилю браузера потенциально может его прочитать). Не предлагать ложное чувство безопасности.
- Data minimization по умолчанию: в AI/n8n уходят только структурированные поля + clean text. Полный HTML — только в debug-режиме, с redaction известных PII-паттернов (email/телефон в тексте).
- n8n webhook: рекомендуется добавить secret header (`X-Signature` или bearer token), который проверяется на стороне n8n workflow — иначе утечка URL webhook (например, через логи) позволяет третьим лицам слать в него произвольные события.
- Никакой телеметрии в адрес разработчика расширения по умолчанию — это прямой коммерческий аргумент в пользу продукта на фоне сервисов автооткликов, которые по факту логинятся под пользователем и парсят сверх необходимого.
- Версионирование схемы IndexedDB с первого дня (Dexie `version()` + миграции) — иначе обновление MVP-1→MVP-2 ломает локальные данные пользователей.

## 11. Risks and Mitigations

| Риск | Вероятность/тяжесть | Митигация |
|---|---|---|
| Бан аккаунта HH за паттерн автоматизации | Высокая тяжесть, растёт после ужесточения дек. 2025 | Core (MVP-1/2) вообще не взаимодействует с формами HH → риск ≈0. Labs (MVP-3/4) — off by default, дневные лимиты, случайные задержки, финальный клик только нативный. |
| Детект synthetic DOM events при автозаполнении | Средняя | Предпочесть clipboard-copy вместо программной записи в поле; если всё же запись в DOM — тестировать на некритичном аккаунте, не на основном. |
| Изменение разметки HH ломает экстракторы | Высокая (типично для любого scraping-инструмента) | DOM fixtures + регулярный re-snapshot, алерт в CI при провале extractor-тестов на свежем снапшоте. |
| Утечка/некорректное использование AI-ключа | Низкая вероятность, высокая тяжесть | Явное предупреждение при онбординге, опция работать вовсе без AI. |
| Ложные срабатывания "сильная вакансия" → спам в Telegram | Средняя на старте | Настраиваемый порог, период калибровки, возможность снизить частоту до дневной сводки вместо realtime. |
| PII третьих лиц в debug HTML | Низкая, но неприятная при обнаружении | Redaction-фильтр даже в debug-режиме. |
| Рассинхрон локального статуса с реальным состоянием на HH | Высокая (пользователь будет действовать и в расширении, и напрямую на сайте) | Пассивное чтение статус-бейджей со страницы HH при каждом визите. |

**Главный вывод по ToS/антибот**: безопасный путь — это не "обойти защиту лучше", а полностью убрать необходимость её обходить. Core продукта работает исключительно как пассивный читатель DOM той страницы, которую пользователь и так открыл сам, плюс локальная аналитика. Это не требует обхода ничего. Risky-часть (semi-auto reply) стоит держать отдельно, маленькой, выключенной по умолчанию, и явно рамить пользователю как "экспериментально, делайте на свой риск, не на основной аккаунт для первых тестов".

## 12. Test Plan

- **Unit-тесты** (vitest): экстракторы (`HHAdapter.extractVacancy/extractSearchList`) на фиксированных DOM-фикстурах; scoring engine на наборе синтетических Job-объектов с ожидаемым score; AI-промпт билдеры (проверка, что constraints maxChars/noEmoji реально встроены в промпт).
- **DOM fixtures**: анонимизированные сохранённые HTML-снапшоты реальных страниц вакансии и поиска HH (без PII). Требуют периодического обновления — разметка HH меняется, это постоянная статья поддержки для любого scraping-инструмента, стоит сразу заложить в roadmap, а не считать одноразовой задачей.
- **E2E** (Playwright): загрузка собранного расширения против локального fixture-сервера, имитирующего разметку HH (не против реального hh.ru — чтобы не создавать лишнюю нагрузку/риск во время CI-прогонов).
- **Ручное тестирование на реальном HH**: небольшой, нечастый smoke-тест на отдельном тестовом аккаунте (не на основном аккаунте, которым пользуетесь для реального поиска работы) — особенно важно для MVP-3/4, где есть прямое взаимодействие с формами.

## 13. Roadmap (соло-разработчик)

- **Спринт 1 (1–2 недели)**: HHAdapter (extractVacancy), Dexie schema v1, rule-based scoring, базовый popup/side panel, ручное изменение статуса.
- **Спринт 2 (1 неделя)**: AI-анализ по кнопке, генерация письма (шаблоны + AI), сохранение финального письма, экспорт CSV/JSON.
- **Спринт 3 (3–5 дней)**: n8n webhook (3 события), Settings/Privacy controls, онбординг-экран с consent.
- **Спринт 4 (1 неделя)**: MVP-2 — extractSearchList, бейджи на карточках, подсветка, быстрые действия.
- **Пауза перед Labs**: использовать MVP-1/2 в реальном поиске работы хотя бы 1–2 недели, прежде чем трогать MVP-3/4 — это даст и реальные данные для калибровки скоринга, и время посмотреть, не изменилась ли политика HH ещё раз.
- **Labs (MVP-3/4)**, только после паузы: clipboard-based подстановка → если стабильно — очередь с лимитами.
- **MVP-5**: после того как Labs подтвердил себя без банов на тестовом аккаунте.

### Финальная рекомендация

Лучший первый MVP — это **MVP-1 без единого байта взаимодействия с формами HH**: чтение, скоринг, письма, трекер, экспорт, события. Это уже самостоятельный продаваемый продукт, он не требует решения сложных вопросов вокруг synthetic events и текущей политики антибота, и его можно безопасно использовать прямо на основном рабочем аккаунте с первого дня. Автоматизация отклика — это V2-история, которую стоит начинать только тогда, когда понятно, как именно сейчас (после декабря 2025) HH детектит automation-паттерны, а не на старте, когда это неизвестно.

## 14. Список конкретных улучшений, которые стоит добавить в ТЗ

1. Явный non-goals список (полная автоотправка, обход капчи, ротация аккаунтов/IP, прямые вызовы api.hh.ru).
2. Онбординг/consent-экран с описанием потоков данных (HH → локально → AI по клику → n8n по событию).
3. Preview "что уйдёт в AI" перед каждым AI-вызовом, с возможностью точечно исключить поля.
4. Маппинг `Resume.hhResumeId` на реальные резюме пользователя на HH, не абстрактная рекомендация.
5. Пассивная ресинхронизация статуса по бейджам со страницы HH (двусторонняя сверка local vs реальность).
6. Настраиваемый порог "сильная вакансия" вместо хардкода 85, плюс период калибровки.
7. Redaction-фильтр для debug-режима с полным HTML, не просто вкл/выкл.
8. Версионирование схемы IndexedDB (Dexie migrations) с первого релиза.
9. Secret header/подпись для n8n webhook.
10. Явное предупреждение о хранении AI-ключа в `chrome.storage.local` как не-secure storage.
11. План поддержки DOM fixtures (регулярный re-snapshot разметки HH, алерты при breaking changes).
12. Explicit kill-switch для Labs-модуля (MVP-3/4) — один тумблер "выключить всё, что взаимодействует с формами HH".

14. Blacklist компаний как переносимый JSON (экспорт/импорт между профилями/устройствами).
15. Чёткое разделение в n8n-событиях MVP-1 vs MVP-5 (`hr_replied` доступен только с MVP-5).
16. Рекомендация тестировать MVP-3/4 на отдельном, не основном HH-аккаунте.
